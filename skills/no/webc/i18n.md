# internasjonalisering

Sak: Se `src/webc/BoxX` for å implementere internasjonalisering av komponenter og demo.

regel:
- `yml` Ikke legg til `:` på slutten av oppføringen, legg inn `:` i pug-malen og js-koden, og spleis den
- Bruk `|-` for flerlinjers tekst av `yml` for enkelt å fjerne etterfølgende mellomrom.
- Hvis det ikke er noen internasjonalisert oppføring, fjern `I18N.js`-filen (ikke skriv noe, ikke skriv `export`)
- Oppføringer må være på ett enkelt språk, og blanding av flere språk er strengt forbudt.

behandle:

1. Se kildespråkkoden som er konfigurert i rotkatalogen `tran.yml` → `tran` → `from`

2. Skriv inn katalogen `src/webc/component name`

3. Internasjonalisere komponenter
   - Skriv teksten som brukes i `komponentnavn.js` som en oppføring i `src/webc/komponentnavn/i18n/kildespråk/js.yml`.
   - Endre `src/webc/component name/I18N.js` og konfigurer konstanten til `yml` oppføringsnøkkelnavn, for eksempel `export const KEY = "ymlKey"`
   - Introduksjon og binding i `src/webc/component name.js` (hvis det er `js.yml`, hvis ikke, er det ikke nødvendig):
     ```js
     import { fetchLang } from "../js/fetchLang.js";
     import i18nUrl, { KEY } fra "./komponentnavn/I18N.js"; // Rammeverket vil automatisk injisere i18nUrl, standard eksportstandard, ikke lag det selv

     const fLang = fetchLang(i18nUrl);
     fLang((i18n) => {
       btn.title = i18n[KEY];
     });
     ```
    Prøv å binde i grupper for å redusere fLang-anrop
    Komponentens `komponentnavn.js` skal ikke ha en streng som inneholder tekst

4. Internasjonalisering av presentasjoner
   - I `demo/_.pug` er etikettteksten som må oversettes i form av `c-t-nøkkelnavn` (dette er en egendefinert etikett, en egen linje), og oppføringen skrives inn i `demo/i18n/kildespråk/pug.yml`
   - `demo/_.js` Skriv den originale tekstoppføringen brukt i `demo/i18n/source language/js.yml` og plasser den i `demo/I18N.js`
   - Introduser `fetchLang` i `demo/_.js` og last inn `./I18N.js` (dette er `demo/I18N.js`), se komponentens internasjonaliseringsprosess ovenfor (for eksempel, `demo/i18n` har ikke `js.yml` nedenfor, hopp over dette trinnet)
   - `demo/I18N.js` Konfigurer nøkkelnavnkonstanter i `demo/i18n/source language/js.yml` og eksporter

5. Skriv komponentbeskrivelsen i `src/webc/component name/i18n/source language/README.md`. Prosessen er som følger
  5.1 Skriv hovedtittelen. Tittelen skal begynne med `#`, beskrive formålet med komponenten i én setning (ikke skriv komponentnavnet), og ikke legg til tegnsetting på slutten.
  5.2 List opp hovedtrekkene til komponenten, start med hovedtittelen, og ikke skriv sekundære titler. Hvis det er flere egenskaper, bruk en uordnet liste (hvis den uordnede listebeskrivelsen er relativt kort, vil det ikke være noen punktum på slutten).
  5.3 Skriv `## Ved å bruke demoen`, starter demokoden med ` ```html ` (i demokoden er enkeltelement-taggen ikke lukket), hvis det er `js`, skriv `><b html` internt`><b>` internt. class="CODE"> Tag, `js` Ikke ha innrykk i den første linjen med innhold.
  Hvis det er ren js, start med ` ```js `. `js` I demonstrasjonen, hvis du trenger å importere `component/_.js`, vennligst bruk `import xxx fra "webc.site/component name.js";`. Dette er den kompilerte `js`, som automatisk vil injisere js og css.
  `js` Bruk kortfattet skrift, for eksempel en `const` for å erklære flere konstanter

Hovedteksten introduserer hvordan du bruker komponenten, grensesnittparametere, veksling mellom `css` og `class` i forskjellige tilstander osv.
Beskriv deretter parametere, returverdi, stilstatus osv. (hvis noen), og legg til en sekundær tittel.

Bruk ord konsist og unngå adjektiver.

Copywriting bør unngå tull og ikke skrive meningsløse ord (som: universal, container, etc.).

6. Kjør `./sh/tran.js` av rotkatalogen, oversettelse
