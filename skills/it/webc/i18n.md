# internazionalizzazione

Caso: fare riferimento a `src/BoxX` per implementare l'internazionalizzazione dei componenti e della demo.

regola:
- `yml` Non aggiungere `:` alla fine della voce, inserisci `:` nel modello pug e nel codice js e uniscilo
- Utilizza `|-` per il testo su più righe di `yml` per rimuovere facilmente gli spazi finali.
- Se non è presente alcuna voce internazionalizzata, cancella il file `I18N.js` (non scrivere nulla, non scrivere `export`)
- Le iscrizioni devono essere in un'unica lingua ed è severamente vietato mischiare più lingue.

processo:

1. Visualizza il codice della lingua sorgente configurato nella directory root `tran.yml` → `tran` → `from`

2. Inserisci la directory `nome src/componente`

3. Internazionalizzare i componenti
   - Scrivi il testo utilizzato da `_.js` come voce in `src/nome componente/i18n/linguaggio sorgente/js.yml`.
   - Modifica `Nome componente/I18N.js` e configura la costante del `yml` nome della chiave di accesso, ad esempio `export const KEY = "ymlKey"`
   - `Introdotto e legato nel componente nome/_.js` (se c'è `js.yml`, altrimenti non è necessario):
     ```js
     import { fetchLang } from "../js/fetchLang.js";
     importa i18nUrl, { CHIAVE } da "./I18N.js"; // Il framework inietterà automaticamente i18nUrl, l'esportazione predefinita, non crearlo tu stesso

     const fLang = fetchLang(i18nUrl);
     fLang((i18n) => {
       btn.title = i18n[KEY];
     });
     ```
    Prova a eseguire il bind in batch per ridurre le chiamate flang
    Il `_.js` del componente non deve contenere stringhe contenenti testo

4. Internazionalizzazione delle presentazioni
   - In `demo/_.pug`, il testo dell'etichetta che deve essere tradotto è occupato sotto forma di `nome chiave c-t` (questa è un'etichetta personalizzata, una riga separata) e la voce è scritta in `demo/i18n/source Language/pug.yml`
   - `demo/_.js` Scrivi la voce di testo originale utilizzata in `demo/i18n/source Language/js.yml` e inseriscila in `demo/I18N.js`
   - Introduci `fetchLang` in `demo/_.js` e carica `./I18N.js` (questo è `demo/I18N.js`), fai riferimento al processo di internazionalizzazione del componente sopra (ad esempio, `demo/i18n` non ha `js.yml` di seguito, salta questo passaggio)
   - `demo/I18N.js` Configura le costanti del nome chiave in `demo/i18n/source Language/js.yml` ed esporta

5. Scrivere la descrizione del componente in `Nome componente/i18n/linguaggio sorgente/README.md`. Il processo è il seguente
  5.1 Scrivi il titolo principale. Il titolo deve iniziare con `#`, descrivere lo scopo del componente in una frase (non scrivere il nome del componente) e non aggiungere segni di punteggiatura alla fine.
  5.2 Elencare le caratteristiche principali del componente, iniziando dal titolo principale, e non scrivere titoli secondari. Se sono presenti più caratteristiche, utilizzare un elenco non ordinato (se la descrizione dell'elenco non ordinato è relativamente breve, non ci sarà alcun punto alla fine).
  5.3 Scrivere `## Usando la demo`, il codice demo inizia con ` ```html ` (nel codice demo, il tag del singolo elemento non è chiuso), se c'è `js`, scrivi `html` internamente, usa il tag `<script type="module">`;
  Se è puro js, ​​inizia con ` ```js `. `js` Nella dimostrazione, se è necessario importare `component/_.js`, utilizzare `import xxx from "webc.site/component name.js";`.
  `js` Utilizza una scrittura concisa, come `const` per dichiarare più costanti

Il testo principale introduce come utilizzare il componente, i parametri dell'interfaccia, il passaggio tra `css` e `class` in diversi stati, ecc.
Successivamente descrivi i parametri, il valore restituito, lo stato dello stile, ecc. (se presente) e aggiungi un titolo secondario.

Usa le parole in modo conciso ed evita gli aggettivi.

Il copywriting dovrebbe evitare sciocchezze e non scrivere parole senza senso (come: universale, contenitore, ecc.).

6. Esegui `./sh/tran.js` della directory root, traduzione
