# internationalisering

Fall: Se `src/BoxX` för att implementera internationalisering av komponenter och demo.

regel:
- `yml` Lägg inte till `:` i slutet av inlägget, lägg in `:` i mopsmallen och js-koden och skarva den
- Använd `|-` för flerradstext av `yml` för att enkelt ta bort efterföljande blanksteg.
- Om det inte finns någon internationaliserad post, rensa `I18N.js`-filen (skriv inget, skriv inte `export`)
- Bidrag måste vara på ett enda språk, och att blanda flera språk är strängt förbjudet.

behandla:

1. Visa källspråkskoden som konfigurerats i rotkatalogen `tran.yml` → `tran` → `from`

2. Ange katalogen `src/component name`

3. Internationalisera komponenter
   - Skriv texten som används av `_.js` som en post i `src/component name/i18n/source language/js.yml`.
   - Ändra `Komponentnamn/I18N.js` och konfigurera konstanten för `yml` inmatningsnyckelnamn, till exempel `export const KEY = "ymlKey"`
   - `Introducerat och bundet i komponentnamn/_.js` (om det finns `js.yml` är det inte nödvändigt om inte):
     ```js
     import { fetchLang } from "../js/fetchLang.js";
     importera i18nUrl, { KEY } från "./I18N.js"; // Ramverket kommer automatiskt att injicera i18nUrl, standardexportstandard, skapa det inte själv

     const fLang = fetchLang(i18nUrl);
     fLang((i18n) => {
       btn.title = i18n[KEY];
     });
     ```
    Försök att binda i omgångar för att minska fLang-anrop
    Komponentens `_.js` bör inte ha strängar som innehåller text

4. Internationalisering av presentationer
   - I `demo/_.pug` är etiketttexten som behöver översättas upptagen i form av `c-t-nyckelnamn` (detta är en anpassad etikett, en separat rad), och posten skrivs in i `demo/i18n/source language/pug.yml`
   - `demo/_.js` Skriv originaltexten som används i `demo/i18n/source language/js.yml` och placera den i `demo/I18N.js`
   - Introducera `fetchLang` i `demo/_.js` och ladda `./I18N.js` (detta är `demo/I18N.js`), se komponentinternationaliseringsprocessen ovan (till exempel, `demo/i18n` har inte `js.yml` nedan, hoppa över det här steget)
   - `demo/I18N.js` Konfigurera nyckelnamnskonstanter i `demo/i18n/source language/js.yml` och exportera

5. Skriv komponentbeskrivningen i `Komponentnamn/i18n/källspråk/README.md`. Processen är som följer
  5.1 Skriv huvudtiteln. Titeln ska börja med `#`, beskriv syftet med komponenten i en mening (skriv inte komponentens namn), och lägg inte till skiljetecken i slutet.
  5.2 Lista huvuddragen för komponenten, börja med huvudtiteln, och skriv inte sekundära titlar. Om det finns flera egenskaper, använd en oordnad lista (om den oordnade listbeskrivningen är relativt kort kommer det inte att finnas någon punkt i slutet).
  5.3 Skriv `## Med hjälp av demon` börjar demokoden med ` ```html ` (i demokoden stängs inte den enskilda elementtaggen), om det finns `js` skriv `><b>html<b">taggen internt, ` internt, använd`html`
  Om det är ren js, börja med ` ```js `. `js` I demonstrationen, om du behöver importera `component/_.js`, använd `import xxx från "webc.site/component/_.js";`.
  `js` Använd kortfattad skrift, till exempel en `const` för att deklarera flera konstanter

Huvudtexten introducerar hur man använder komponenten, gränssnittsparametrar, växling mellan `css` och `class` i olika tillstånd, etc.
Beskriv sedan parametrarna, returvärde, stilstatus, etc. (om någon) och lägg till en sekundär titel.

Använd ord kortfattat och undvik adjektiv.

Copywriting bör undvika nonsens och inte skriva meningslösa ord (som: universal, container, etc.).

6. Kör `./sh/tran.js` av rotkatalogen, översättning
