# kansainvälistyminen

Case: Katso `src/webc/BoxX` toteuttaaksesi komponenttien kansainvälistämisen ja esittelyn.

sääntö:
- `yml` Älä lisää `:` merkinnän loppuun, laita `:` pug-malliin ja js-koodiin ja liitä se
- Käytä `|-`-merkkiä `yml`:n moniriviselle tekstille, jotta voit helposti poistaa välilyönnit.
- Jos kansainvälistä merkintää ei ole, tyhjennä `I18N.js`-tiedosto (älä kirjoita mitään, älä kirjoita `export`)
- Osallistujien on oltava yhdellä kielellä, ja useiden kielten sekoittaminen on ehdottomasti kielletty.

käsitellä:

1. Tarkastele juurihakemistoon `tran.yml` → `tran` → `from` määritettyä lähdekielen koodia

2. Kirjoita `src/webc/component name` -hakemisto

3. Kansainvälistää komponentteja
   - Kirjoita `komponentin nimi.js`:ssa käytetty teksti merkinnäksi tiedostoon `src/webc/komponentin nimi/i18n/source language/js.yml`.
   - Muokkaa `src/webc/komponentin nimi/I18N.js` ja määritä `yml` syöttöavaimen nimen vakio, kuten `export const KEY = "ymlKey"`
   - Johdanto ja sidonta tiedostossa `src/webc/component name.js` (jos on `js.yml`, jos ei, se ei ole välttämätöntä):
     ```js
     import { fetchLang } from "../js/fetchLang.js";
     tuo i18nUrl, { KEY } kohteesta "./komponentin nimi/I18N.js"; // Kehys lisää automaattisesti i18nUrl:n, oletusarvoisen viennin, älä luo sitä itse

     const fLang = fetchLang(i18nUrl);
     fLang((i18n) => {
       btn.title = i18n[KEY];
     });
     ```
    Yritä sitoa erissä fLang-kutsujen vähentämiseksi
    Komponentin `komponentin nimi.js` ei saa sisältää tekstiä sisältävää merkkijonoa

4. Esitysten kansainvälistyminen
   - Kohdassa `demo/_.pug` käännettävä tunnisteteksti on varattu muodossa `c-t-avaimen nimi` (tämä on mukautettu tunniste, erillinen rivi), ja merkintä kirjoitetaan tiedostoon `demo/i18n/source language/pug.yml`
   - `demo/_.js` Kirjoita alkuperäinen teksti, jota käytetään tiedostossa `demo/i18n/source language/js.yml` ja sijoita se kohtaan `demo/I18N.js`
   - Lisää `fetchLang` kohtaan `demo/_.js` ja lataa `./I18N.js` (tämä on `demo/I18N.js`), katso yllä olevaa komponenttien kansainvälistymisprosessia (esimerkiksi kohdassa `demo/i18n` ei ole `js.yml` alla, ohita tämä vaihe)
   - `demo/I18N.js` Määritä avainten nimivakiot tiedostossa `demo/i18n/source language/js.yml` ja vie

5. Kirjoita komponentin kuvaus tiedostoon `src/webc/komponentin nimi/i18n/source language/README.md`. Prosessi on seuraava
  5.1 Kirjoita pääotsikko. Otsikon tulee alkaa kirjaimella `#`, kuvailla komponentin tarkoitus yhdellä lauseella (älä kirjoita komponentin nimeä), äläkä lisää välimerkkejä loppuun.
  5.2 Listaa komponentin pääominaisuudet alkaen pääotsikosta, äläkä kirjoita toissijaisia ​​nimikkeitä. Jos ominaisuuksia on useita, käytä järjestämätöntä listaa (jos järjestämättömän luettelon kuvaus on suhteellisen lyhyt, lopussa ei ole pistettä).
  5.3 Kirjoita `## Demoa käytettäessä` esittelykoodi alkaa ` ```html ` (esittelykoodissa yksielementtitunniste ei ole suljettu), jos on `js`, kirjoita `html` Tag="CODE"> ><b" class="CODE"> sisäisesti, käytä `js` Älä sisällä sisennystä ensimmäisellä sisältörivillä.
  Jos se on puhdas js, aloita ` ```js `. `js` Jos sinun on tuotava `component/_.js`, käytä esittelyssä komentoa `import xxx osoitteesta "webc.site/component name.js";`. Tämä on käännetty `js`, joka lisää automaattisesti js:n ja css:n.
  `js` Käytä tiivistä kirjoitusta, kuten `const` ilmoittaaksesi useita vakioita

Pääteksti esittelee komponentin käyttöä, käyttöliittymäparametreja, vaihtamista `css` ja `class` välillä eri tiloissa jne.
Kuvaile sitten parametrit, palautusarvo, tyylin tila jne. (jos sellainen on) ja lisää toissijainen otsikko.

Käytä sanoja ytimekkäästi ja vältä adjektiiveja.

Tekstinkirjoittamisessa tulee välttää hölynpölyä äläkä kirjoita merkityksettömiä sanoja (kuten: universaali, kontti jne.).

6. Suorita juurihakemiston `./sh/tran.js` käännös
