# 國際化

案例: 參考 `src/webc/BoxX` 實現元件與 Demo 的國際化。

規則：
- `yml` 詞條不在結尾加上 `:`，把 `:` 放到 pug 範本、js 程式碼中，拼接
- `yml` 的多行文字用 `|-`，方便去掉結尾空格
- 如無國際化詞條，清空 `I18N.js` 檔（什麼內容不要寫，不要寫 `export`）
- 詞條用單一語言，嚴禁多語言混雜

流程:

1. 查看根目錄下 `tran.yml` → `tran` → `from` 配置的原始語言代碼

2. 進入 `src/webc/組件名`目錄

3. 對組件國際化
   - 將 `元件名.js` 用到的文本，作為詞條寫入 `src/webc/元件名/i18n/來源語言/js.yml`。
   - 修改 `src/webc/元件名/I18N.js`，配置 `yml` 詞條鍵名的常數，如 `export const KEY = "ymlKey"`
   - `src/webc/元件名.js` 中引入並綁定（如有 `js.yml`，沒有不需要）：
     ```js
     import { fetchLang } from "../js/fetchLang.js";
     import i18nUrl, { KEY } from "./元件名/I18N.js"; // 框架會自動注入 i18nUrl 這個預設的 export default，不要自己創建

     const fLang = fetchLang(i18nUrl);
     fLang((i18n) => {
       btn.title = i18n[KEY];
     });
     ```
    盡量批量綁定，減少 fLang 的調用
    元件的 `元件名.js` 不要有包含文字的字串

4. 示範的國際化
   - `demo/_.pug` 中，需要翻譯的標籤文字透過 `c-t 鍵名` 形式佔位（這是自訂標籤，單獨一行 )，並將詞條寫入 `demo/i18n/來源語言/pug.yml`
   - `demo/_.js` 用到的原文詞條寫入 `demo/i18n/源語言/js.yml`，並在 `demo/I18N.js`
   - `demo/_.js` 中引入 `fetchLang` 載入 `./I18N.js` （這是 `demo/I18N.js`），參考上面的元件國際化流程（如 `demo/i18n` 下面沒有 `js.yml`，跳過這一步）
   - `demo/I18N.js` 配置 `demo/i18n/源語言/js.yml` 中的鍵名常量，導出

5. 在 `src/webc/組件名稱/i18n/來源語言/README.md` 寫入組件說明，流程如下
  5.1 寫主標題， 標題以 `#` 開頭，一句話描述元件用途 (不寫元件名稱)，結尾不加標點。
  5.2 列出組件的主要特性，接著主標題開始，不寫二級標題。如有多條特性，用無序表(無序表描述如果比較短，結尾不加句號)。
  5.3 寫`## 使用示範`，示範程式碼以 ` ```html ` 開頭(示範程式碼中，單元素標籤不加閉合)，如有 `js`，寫 `html<b class="b class="CODE">，寫 `標籤，`js` 內容首行不要有縮排。
  如果是純 js，用` ```js `開頭。 `js` 示範中，如需導入 `元件/_.js`，請用 `import xxx from "webc.site/元件名稱.js";`，這是編譯後的 `js`，會自動注入 js 和 css。
  `js` 用簡潔的寫法，例如一個 `const` 宣告多個常數

正文介紹元件如何使用、介面參數、不同狀態 `css` 的 `class` 的切換等。
後續描述參數、回傳值、樣式狀態等(如有)，加二級標題。

用詞要簡潔，不寫形容詞。

文案要避免廢話，不寫沒有意義的字（例如：通用、容器等等）。

6. 運行根目錄的 `./sh/tran.js`，翻譯
