# WebC.site

## What's this?

### This is a native web component library

Developed based on [web components](https://developer.mozilla.org/docs/Web/API/Web_components), it is runtime independent and suitable for all frameworks.

Extremely optimized for size (one-tenth the size of similar popular libraries).

Supports 75 languages. You can directly import js and css for online use, or add components to your project as needed for secondary development.

<img src="https://i-01.eu.org/20260604114250265.avif" width="400">

### This is the component library development framework

This is the static blog version of [storybook](https://storybook.js.org).

Building a component library can improve the company's internal development efficiency and build a brand image in the technical circle.

Based on [webc.site](https://github.com/webc-site/webc.site), teams can easily create their own component libraries and publish them to [github page](https://pages.github.com), [cloudflare page](https://pages.cloudflare.com), etc. The published components can be debugged online on [codepen.io](https://codepen.io) and [jsfiddle](https://jsfiddle.net).

At the same time, the built-in translation framework can quickly realize the internationalization of components and documents, so that your product audience is no longer bound by language.

### This is a new paradigm of front-end development for the era of artificial intelligence

The framework is built-in `.agents/skills`, allowing artificial intelligence to help you develop new components with one click according to best practices.

<img src="https://i-01.eu.org/20260604121408079.avif" width="400">

Let the front-end code change from an unsustainable garbage mountain of atmosphere programming into a standard part that can be maintained and iteratively evolved over the long term.

In view of the characteristics of artificial intelligence-assisted development, we proposed and designed a new front-end development paradigm:

1. **Everything is a web component**
2. **The component must be independent of the back-end data link, and the data interface is exposed as a callback function**

In this way, artificial intelligence can generate fake data and display different states of components on the demo page.

Separation of component development and data interface, combined with [webc.site component framework](https://github.com/webc-site/webc.site) Another major feature:

**Each component can start the development server independently and develop and debug on the demo page**.

<img src="https://i-01.eu.org/webcSiteDemo.svg" width="400">
<img src="https://i-01.eu.org/20260604112614499.avif" width="400">

This reduces the front-end link for development and debugging of each component to `0`.

In this way, artificial intelligence-assisted development tools ([Claude Code](https://code.claude.com), [OpenAI Codex](https://openai.com), [Google Antigravity](https://antigravity.google), [Cursor](https://cursor.com), [Windsurf](https://windsurf.com), [Devin](https://devin.ai), [OpenHands](https://github.com/All-Hands-AI/OpenHands) etc.), you can enjoy fully automatic development, open browser debugging, and no longer be blocked by various messy factors (backend data, login status, path routing, etc.).

If you want to control artificial intelligence and reduce costs and increase efficiency, you need a new paradigm that minimizes the development and debugging environment of each component.

### Web component library

#### Ultimate size optimization

For example, for the virtual scroll bar component, compare the size of the popular [OverlayScrollbars](https://github.com/KingSora/OverlayScrollbars) script and style after `gz`, as shown in the following table:

|                   |   js    |  css   |     total      |
| :---------------- | :-----: | :----: | :------------: |
| overlayscrollbars | 14520 B | 2584 B | 17104 B (9.1X) |
| Scroll            | 1159 B  | 721 B  |   ⭐ 1880 B    |

`webc.site`'s virtual scroll bar component is only `OverlayScrollbars`'s `10%`.

#### Runtime independent

Built on standards [Web Components](https://developer.mozilla.org/docs/Web/API/Web_components), the webc.site component library is naturally runtime-agnostic. Whether you are using React, Vue, Angular, Svelte, or a traditional pure HTML/JS project, all components can run directly in the browser like native HTML tags, without the need for secondary packaging for a specific framework.

##### Comparison with shadcn

[shadcn/ui](https://ui.shadcn.com) adopts the code distribution mode, which requires you to copy the component code directly into your own project. It is tightly bound to the React ecosystem and specific construction tool chains (such as Tailwind CSS), and cannot be used across frameworks. It also has certain requirements for project engineering configuration.

And the component of `webc.site`:
- **Cross-framework universal**: Write once and can be directly introduced and run in any front-end framework (even a framework-free environment).
- **Environment zero dependency**: No specific packaging tools or CSS preprocessors are required, just import the compiled static files and it can be used out of the box.

## Automated development and debugging

### Local development and debugging

Each web page component supports starting the development server independently, providing a minimal development and debugging environment:

- **Create a new component**: Run `./sh/new.js [css|js] [component name]` to quickly generate a component development skeleton (including demo page and internationalization configuration).
- **Independent debugging**: Run `./dev.js [component name]` (or run `./dev.js` directly) to start the Vite development server and automatically open the demo page of the component in the browser. Supports component hot update (HMR). Because development and data links are decoupled, there is zero development front-end link, which is very suitable for AI (such as Claude Code, Google Antigravity, etc.) to assist in fully automatic development and browser debugging.

### Publish component library

#### webc dist

The packaging and publishing process is highly automated:
- **Static resource packaging**: Running `./sh/dist.js` (that is, executing the `webc dist` task) can automatically scan all web page components under `src/`, compile, compress and output them to the `dist/` directory. This directory contains JS/CSS files for production distribution, a demo site ready for deployment to GitHub Pages or Cloudflare Pages, and automatically generated `_redirects` and `404.html` routing files.
- **NPM Publish**: Run `./sh/npmDist.js`, the script will automatically handle component dependencies, increment the package version number, and execute `npm publish` with one click after packaging is completed to publish the component library to NPM.

### The platform is responsible for translation

The framework has built-in automated translation processes:
1. Configure translation rules (such as source language, target language and directory to be translated) in `tran.yml` in the project root directory.
2. Use placeholders in components or documents without having to hardcode multiple languages ​​in your code.
3. Run `./sh/tran.js`, and the translation engine will automatically extract the text and call the WebC translation service for processing, automatically generating YAML translation entries and README documentation in various languages.

### Environment Variables

When performing automated translation, you need to configure the following environment variables to authorize access to the translation API:

- **`WEBC_SITE_TOKEN`** (or **`WEBC_TOKEN`**): Authentication key used to access the `webc.site` Automated Translation API. Can be injected as an environment variable or written `token` in the `~/.config/webc.site.yml` configuration file in the local user's home directory.
