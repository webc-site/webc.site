# WebC.site

## これは何ですか？

### これはネイティブ Web コンポーネント ライブラリです

[Web コンポーネント](https://developer.mozilla.org/docs/Web/API/Web_components) に基づいて開発されており、ランタイムに依存せず、すべてのフレームワークに適しています。

サイズが非常に最適化されています (同様の一般的なライブラリのサイズの 10 分の 1)。

75の言語をサポートします。オンラインで使用するために js と css を直接インポートしたり、二次開発の必要に応じてプロジェクトにコンポーネントを追加したりできます。

<img src="https://i-01.eu.org/20260604114250265.avif" width="400">

### これはコンポーネント ライブラリの開発フレームワークです

[storybook](https://storybook.js.org) の静的ブログ版です。

コンポーネントライブラリを構築すると、社内の開発効率が向上し、技術界でのブランドイメージを構築できます。

[webc.site](https://github.com/webc-site/webc.site) に基づいて、チームは独自のコンポーネント ライブラリを簡単に作成し、[github ページ](https://pages.github.com)、[cloudflare ページ](https://pages.cloudflare.com) などに公開できます。公開されたコンポーネントは、[codepen.io](https://codepen.io) および [jsfiddle](https://jsfiddle.net) でオンラインでデバッグできます。

同時に、組み込みの翻訳フレームワークにより、コンポーネントとドキュメントの国際化を迅速に実現できるため、製品の視聴者は言語に縛られることがなくなります。

### これは人工知能時代のフロントエンド開発の新しいパラダイムです

フレームワークが`.agents/skills`組み込まれているため、人工知能がベスト プラクティスに従ってワンクリックで新しいコンポーネントを開発できるようになります。

<img src="https://i-01.eu.org/20260604121408079.avif" width="400">

フロントエンドのコードを、持続不可能な雰囲気プログラミングのゴミの山から、長期にわたって維持および反復的に進化できる標準的な部分に変えましょう。

人工知能支援開発の特性を考慮して、私たちは新しいフロントエンド開発パラダイムを提案および設計しました。

1. **すべては Web コンポーネントです**
2. **コンポーネントはバックエンド データ リンクから独立している必要があり、データ インターフェイスはコールバック関数として公開されます**

このようにして、人工知能は偽のデータを生成し、デモ ページにコンポーネントのさまざまな状態を表示できます。

コンポーネント開発とデータ インターフェイスの分離と [webc.site コンポーネント フレームワーク](https://github.com/webc-site/webc.site) の組み合わせ もう 1 つの主要な機能:

**各コンポーネントは独立して開発サーバーを起動し、デモ ページで開発とデバッグを行うことができます**。

<img src="https://i-01.eu.org/webcSiteDemo.svg" width="400">
<img src="https://i-01.eu.org/20260604112614499.avif" width="400">

これにより、各コンポーネントの開発とデバッグのためのフロントエンド リンクが`0`に削減されます。

このようにして、人工知能支援開発ツール ([Claude Code](https://code.claude.com)、[OpenAI Codex](https://openai.com)、[Google Antigravity](https://antigravity.google)、[Cursor](https://cursor.com)、[Windsurf](https://windsurf.com)、[Devin](https://devin.ai)、[OpenHands](https://github.com/All-Hands-AI/OpenHands) など) を使用すると、完全自動開発、オープン ブラウザ デバッグを楽しむことができ、さまざまな厄介な要因によってブロックされることがなくなります。 (バックエンド データ、ログイン ステータス、パス ルーティングなど)。

人工知能を制御してコストを削減し、効率を向上させたい場合は、各コンポーネントの開発およびデバッグ環境を最小限に抑える新しいパラダイムが必要です。

`webc.site` の仮想スクロール バー コンポーネントは `OverlayScrollbars` の `10%` のみです。

#### ランタイムに依存しない

標準 [Web コンポーネント](https://developer.mozilla.org/docs/Web/API/Web_components) に基づいて構築された webc.site コンポーネント ライブラリは、当然ながらランタイムに依存しません。 React、Vue、Angular、Svelte、または従来の純粋な HTML/JS プロジェクトを使用しているかどうかに関係なく、すべてのコンポーネントはネイティブ HTML タグのようにブラウザーで直接実行でき、特定のフレームワーク用の二次パッケージ化は必要ありません。

##### shadcnとの比較

[shadcn/ui](https://ui.shadcn.com) はコード配布モードを採用しているため、コンポーネント コードを独自のプロジェクトに直接コピーする必要があります。 React エコシステムと特定の構築ツール チェーン (Tailwind CSS など) に密接にバインドされており、フレームワークを越えて使用することはできません。また、プロジェクトのエンジニアリング構成にも特定の要件があります。

そして`webc.site`のコンポーネント:
- **クロスフレームワーク ユニバーサル**: 一度作成すれば、任意のフロントエンド フレームワーク (フレームワークのない環境でも) に直接導入して実行できます。
- **環境依存性なし**: 特定のパッケージ化ツールや CSS プリプロセッサは必要ありません。コンパイルされた静的ファイルをインポートするだけで、すぐに使用できます。

## 自動化された開発とデバッグ

### ローカル開発とデバッグ

各 Web ページ コンポーネントは開発サーバーの独立した起動をサ​​ポートし、最小限の開発およびデバッグ環境を提供します。

- **新しいコンポーネントの作成**: `./sh/new.js [css|js] [コンポーネント名]` を実行して、コンポーネント開発スケルトン (デモ ページと国際化構成を含む) をすばやく生成します。
- **独立デバッグ**: `./dev.js [コンポーネント名]` を実行 (または `./dev.js` を直接実行) して Vite 開発サーバーを起動し、ブラウザでコンポーネントのデモ ページを自動的に開きます。コンポーネントのホット アップデート (HMR) をサポートします。開発とデータのリンクが分離されているため、開発フロントエンド リンクはゼロです。これは、完全に自動化された開発とブラウザのデバッグを支援する AI (Claude Code、Google Antigravity など) に非常に適しています。

### コンポーネントライブラリを公開する

#### webc dist

パッケージ化と公開のプロセスは高度に自動化されています。
- **静的リソースのパッケージ化**: `./sh/dist.js` を実行すると (つまり、`webc dist` タスクを実行すると)、`src/` の下にあるすべての Web ページ コンポーネントが自動的にスキャンされ、コンパイル、圧縮されて、`dist/` ディレクトリに出力されます。このディレクトリには、実稼働配布用の JS/CSS ファイル、GitHub Pages または Cloudflare Pages にデプロイする準備ができているデモ サイト、および自動生成された`_redirects`および`404.html`ルーティング ファイルが含まれています。
- **NPM 発行**: `./sh/npmDist.js` を実行すると、スクリプトがコンポーネントの依存関係を自動的に処理し、パッケージのバージョン番号を増分し、パッケージ化の完了後にワンクリックで `npm publish` を実行してコンポーネント ライブラリを NPM に発行します。

### プラットフォームが翻訳を担当します

フレームワークには自動翻訳プロセスが組み込まれています。
1. プロジェクトのルート ディレクトリの `tran.yml` で翻訳ルール (ソース言語、ターゲット言語、翻訳対象のディレクトリなど) を設定します。
2. コード内で複数の言語をハードコーディングすることなく、コンポーネントまたはドキュメントでプレースホルダーを使用できます。
3. `./sh/tran.js` を実行すると、翻訳エンジンがテキストを自動的に抽出し、WebC 翻訳サービスを呼び出して処理し、YAML 翻訳エントリと README ドキュメントをさまざまな言語で自動的に生成します。

### 環境変数

自動翻訳を実行する場合、翻訳 API へのアクセスを承認するには、次の環境変数を構成する必要があります。

- **`WEBC_SITE_TOKEN`** (または **`WEBC_TOKEN`**): `webc.site` 自動翻訳 API へのアクセスに使用される認証キー。環境変数として挿入することも、ローカル ユーザーのホーム ディレクトリの`~/.config/webc.site.yml`設定ファイルに`token`書き込むこともできます。
