# Спецификация JS-кода

## стиль кодирования

- Просто, элегантно и эффективно, с использованием самого современного синтаксиса js.
- Разделите слишком длинные функции, а дизайн интерфейса должен иметь низкую связанность и высокую связность.
- Больше используйте массивы + циклы/forEach/map и пишите меньше повторяющегося кода.
- Не пишите комментарии к функциям в простом коде, чтобы избежать излишних комментариев. Сложная логика и особые потребности объясняются на изысканном китайском языке.
- Повторяющийся код абстрагируется на функции для уменьшения избыточности. Например, `cosnt A=new Table({ style: { border: NO_BORDER } }), B=new Table({ style: { border: NO_BORDER } })` можно записать как `const newTable=()=>new Table({ style: { border: NO_BORDER } }), A=newTable(), B=newTable()`
- Объявление переменной: объедините несколько последовательных объявлений `const` в один оператор. Вместо того, чтобы писать это в три строки, напишите `const a=1, b=2, c=3;`
- Асинхронная обработка: используйте `await`, запретите использование цепных вызовов `.then`
- Не генерировать автоматически код обработки исключений, не записывать автоматически `try...catch` (`try ... catch` поддерживается вручную, существующий `try catch` не следует удалять)
- Сначала чистые функции: пишите только чистые функции, а не классы.
- Используйте стрелочные функции `const funcName = () => {}` и не используйте ключевое слово `function` (кроме генераторов); если вы можете использовать .bind для привязки параметров, избегайте определения функций
- Повторное использование кода. Обратите внимание на повторное использование, извлекайте больше мелких функций и решительно избегайте большого количества похожих или скопированных структур кода.
- Доступ к объекту: расставьте приоритеты с использованием назначения деструктуризации для извлечения необходимых атрибутов, избегайте многократного внутреннего использования точек для доступа к глубоким и вложенным атрибутам и объединяйте повторяющиеся необязательные цепочки решений.
- Не используйте объекты для параметров функции, не пишите `{a,b,c}`, пишите `a, b, c`; если имеется много необязательных параметров, используйте [[количество элементов конфигурации, элементы конфигурации], [количество элементов конфигурации, элементы конфигурации]...] В этой парадигме элементы конфигурации определяются с помощью числовых констант.
- Для возврата нескольких значений используйте массив `[a,b,c]` вместо `{a,b,c}`. Если возвращается более трех значений, используйте числовые константы для определения позиционного значения.
- Нет необходимости в строке для представления состояния, используйте числовые константы для определения состояния.
- Шаблон строки (``) не используется, используется конкатенация строк (+), за исключением импорта (удобно для статического анализа Vite)
- Для цикла, если вам нужен серийный номер, используйте `++i` вместо `i++`
- Избегайте ненужного`?.` и запретите защитное программирование.
- Имя переменной не обязательно должно быть во множественном числе и заканчивается на `_li`, обозначая список.

## Соглашение об именах

- Название преследует минимализм. Используйте короткие, но содержательные имена, например: используйте `rm` вместо `remove`, `delete`, `del`. Однако вам также следует избегать крайностей, например: не используйте одну букву `m` вместо `map`.
- Старайтесь использовать в именовании функций только глаголы. Если вы можете выразить это одним словом, не используйте два слова. Существительные отражаются в именах файлов. Если вам нужно добавить глагол к имени файла, сначала поставьте существительное, а затем глагол. Например: `profileSet.js` вместо `setProfile.js`
- Имя переменной: используйте стиль подчеркивания (snake_case), например `user_auth_token`; если переменная является функцией, используйте стиль именования в нижнем регистре
- Имя функции: используйте CamelCase в нижнем регистре (camelCase), например `userData`
- Параметры функции: если это функция обратного вызова, назовите ее строчными буквами, например `onChange`
- В определениях констант на уровне модуля используется подчеркивание в верхнем регистре `UPPER_SNAKE_CASE`
- Не пишите `get`, такие бессмысленные префиксы, как: пишите `cookieByHeader` вместо `getCookie`
- Глобальные константы/константы уровня модуля: используйте стиль подчеркивания заглавными буквами (UPPER_SNAKE_CASE), например `CODE_TO_ID`, `ID_TO_LANG`

## Модульный механизм

- Импорт: точно импортируйте функции по требованию, запретите прямой импорт всего модуля (избегайте `import * as x` или импортируйте большие объекты).
- Экспорт: отключить экспорт объектов. Экспортируйте функции и переменные как единицы измерения, например `export const X=1, abc=()=>{};`. Попробуйте объединить их с помощью const + запятой, чтобы объявить экспортируемый контент. Если файл имеет только одну функцию, используйте `export default`
- Если вам не требуется внутренний вызов функции экспорта по умолчанию, избегайте сначала объявления констант, а затем их экспорта в конец файла.
- Разрешение пути: при получении текущего пути к каталогу необходимо использовать `import.meta.dirname`.

## ошибка

- Избегайте строковых ошибок и используйте const для объявления постоянных кодов ошибок.
- Если вам нужно вернуть неверную информацию о данных, используйте [код ошибки, сообщение об ошибке,...]. Таким образом, сообщение об ошибке представляет собой не текстовое описание, а числовое значение (например, [FILE_OVERSIZE, file_size, max_size]).
