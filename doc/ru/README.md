# WebC.site

## Что это?

### Это собственная библиотека веб-компонентов.

Разработанный на основе [веб-компонентов](https://developer.mozilla.org/docs/Web/API/Web_components), он не зависит от среды выполнения и подходит для всех платформ.

Чрезвычайно оптимизирован по размеру (одна десятая размера аналогичных популярных библиотек).

Поддерживает 75 языков. Вы можете напрямую импортировать js и css для использования в Интернете или добавлять компоненты в свой проект по мере необходимости для вторичной разработки.

### Это среда разработки библиотеки компонентов.

Это статическая версия блога [сборника рассказов](https://storybook.js.org).

Создание библиотеки компонентов может повысить эффективность внутренней разработки компании и создать имидж бренда в технических кругах.

На основе [webc.site](https://github.com/webc-site/webc.site) команды могут легко создавать свои собственные библиотеки компонентов и публиковать их на [странице github](https://pages.github.com), [странице Cloudflare](https://pages.cloudflare.com) и т. д. Опубликованные компоненты можно отлаживать онлайн на [codepen.io](https://codepen.io) и [jsfiddle](https://jsfiddle.net).

В то же время встроенная система перевода может быстро реализовать интернационализацию компонентов и документов, так что аудитория вашего продукта больше не будет привязана к языку.

### Это новая парадигма фронтенд-разработки эпохи искусственного интеллекта.

В платформу встроен `.agents/skills`, позволяющий искусственному интеллекту помочь вам разрабатывать новые компоненты одним щелчком мыши в соответствии с передовыми практиками.

<img src="https://i-01.eu.org/20260604121408079.avif" width="400">

Позвольте интерфейсному коду превратиться из неустойчивой горы мусора атмосферного программирования в стандартную часть, которую можно поддерживать и итеративно развивать в течение длительного времени.

Учитывая особенности разработки с использованием искусственного интеллекта, мы предложили и разработали новую парадигму фронтенд-разработки:

1. **Все является веб-компонентом**
2. **Компонент должен быть независимым от внутреннего канала передачи данных, а интерфейс данных предоставляется как функция обратного вызова**

Таким образом, искусственный интеллект может генерировать фейковые данные и отображать на демонстрационной странице разные состояния компонентов.

Разделение разработки компонентов и интерфейса данных в сочетании с [инфраструктурой компонентов webc.site](https://github.com/webc-site/webc.site). Еще одна важная особенность:

**Каждый компонент может запускать сервер разработки независимо, а также разрабатывать и отлаживать его на демонстрационной странице**.

<img src="https://i-01.eu.org/webcSiteDemo.svg" width="400">

Это сокращает количество внешних ссылок для разработки и отладки каждого компонента до `0`.

Таким образом, с помощью инструментов разработки с использованием искусственного интеллекта ([Claude Code](https://code.claude.com), [OpenAI Codex](https://openai.com), [Google Antigravity](https://antigravity.google), [Cursor](https://cursor.com), [Windsurf](https://windsurf.com), [Devin](https://devin.ai), [OpenHands](https://github.com/All-Hands-AI/OpenHands) и т. д.), вы можете наслаждаться полностью автоматической разработкой, открытой отладкой браузера и больше не блокироваться различными неприятными факторами (серверные данные, статус входа в систему, маршрутизация путей и т. д.).

Если вы хотите контролировать искусственный интеллект, сокращать затраты и повышать эффективность, вам нужна новая парадигма, которая сводит к минимуму среду разработки и отладки каждого компонента.

#### Независимый от времени выполнения

Построенная на основе стандартов [Веб-компоненты](https://developer.mozilla.org/docs/Web/API/Web_components), библиотека компонентов webc.site, естественно, не зависит от времени выполнения. Независимо от того, используете ли вы React, Vue, Angular, Svelte или традиционный проект на чистом HTML/JS, все компоненты могут запускаться непосредственно в браузере, как собственные HTML-теги, без необходимости вторичной упаковки для конкретной платформы.

##### Сравнение с shadcn

[shadcn/ui](https://ui.shadcn.com) использует режим распространения кода, который требует от вас копирования кода компонента непосредственно в ваш собственный проект. Он тесно связан с экосистемой React и конкретными цепочками инструментов построения (такими как Tailwind CSS) и не может использоваться в разных платформах. Он также имеет определенные требования к инженерной конфигурации проекта.

И компонент `webc.site`:
- **Универсальность для разных платформ**: напишите один раз, и его можно напрямую внедрить и запустить в любой внешней среде (даже в среде без фреймворка).
- **Нулевая зависимость от среды**: никаких специальных инструментов упаковки или препроцессоров CSS не требуется, просто импортируйте скомпилированные статические файлы, и их можно будет использовать сразу после установки.

## Автоматизированная разработка и отладка

### Локальная разработка и отладка

Каждый компонент веб-страницы поддерживает независимый запуск сервера разработки, обеспечивая минимальную среду разработки и отладки:

- **Создайте новый компонент**. Запустите `./sh/new.js [css|js] [имя компонента]`, чтобы быстро создать скелет разработки компонента (включая демонстрационную страницу и конфигурацию интернационализации).
- **Независимая отладка**: запустите `./dev.js [имя компонента]` (или запустите `./dev.js` напрямую), чтобы запустить сервер разработки Vite и автоматически открыть демонстрационную страницу компонента в браузере. Поддерживает горячее обновление компонентов (HMR). Поскольку связи между разработкой и данными разъединены, отсутствует связь с внешним интерфейсом разработки, что очень подходит для ИИ (например, Claude Code, Google Antigravity и т. д.) для помощи в полностью автоматической разработке и отладке браузера.

### Публикация библиотеки компонентов

#### webc dist

Процесс упаковки и публикации высокоавтоматизирован:
- **Упаковка статических ресурсов**: запуск `./sh/dist.js` (то есть выполнение задачи `webc dist`) может автоматически сканировать все компоненты веб-страницы в `src/`, компилировать, сжимать и выводить их в каталог `dist/`. Этот каталог содержит файлы JS/CSS для производственного распространения, демонстрационный сайт, готовый к развертыванию на страницах GitHub или Cloudflare, а также автоматически созданные файлы маршрутизации `_redirects` и `404.html`.
- **Публикация NPM**: запустите `./sh/npmDist.js`, сценарий автоматически обработает зависимости компонентов, увеличит номер версии пакета и выполнит `npm publish` одним щелчком мыши после завершения упаковки, чтобы опубликовать библиотеку компонентов в NPM.

### Платформа отвечает за перевод

Фреймворк имеет встроенные процессы автоматического перевода:
1. Настройте правила перевода (например, исходный язык, целевой язык и каталог для перевода) в `tran.yml` в корневом каталоге проекта.
2. Используйте заполнители в компонентах или документах без необходимости жесткого кодирования нескольких языков в своем коде.
3. Запустите `./sh/tran.js`, и механизм перевода автоматически извлечет текст и вызовет службу перевода WebC для обработки, автоматически создав записи перевода YAML и документацию README на различных языках.

### переменные среды

При выполнении автоматического перевода вам необходимо настроить следующие переменные среды, чтобы авторизовать доступ к API перевода:

- **`WEBC_SITE_TOKEN`** (или **`WEBC_TOKEN`**): ключ аутентификации, используемый для доступа к API автоматического перевода `webc.site`. Может быть внедрен как переменная среды или записан `token` в файле конфигурации `~/.config/webc.site.yml` в домашнем каталоге локального пользователя.
