# Специфікація коду JS

## стиль кодування

- Простий, елегантний і ефективний, використовуючи найсучасніший синтаксис js
- Розділіть надто довгі функції, а дизайн інтерфейсу має бути низьким зв’язком і високою зв’язністю.
- Більше використовуйте масиви + цикли/forEach/map і пишіть менше повторюваного коду
- Не пишіть коментарі до функцій у простому коді, щоб уникнути надмірних коментарів. Складна логіка та особливі потреби пояснюються вишуканою китайською.
- Повторюваний код абстрагується у функції, щоб зменшити надмірність. Наприклад, `cosnt A=new Table({ style: { border: NO_BORDER } }), B=new Table({ style: { border: NO_BORDER } })` можна записати як `const newTable=()=>new Table({ style: { border: NO_BORDER } }), A=newTable(), B=newTable()`
- Оголошення змінної: об’єднайте кілька послідовних `const` декларацій в один оператор. Замість написання в три рядки напишіть `const a=1, b=2, c=3;`
- Асинхронна обробка: використовувати `await`, заборонити використання ланцюжкових викликів `.then`
- Не створювати автоматично код обробки винятків, не записувати автоматично `try...catch` (`try ... catch` підтримується вручну, існуючий `try catch` не слід видаляти)
- Спочатку чисті функції: пишіть лише чисті функції, а не класи
- Використовуйте функції зі стрілками `const funcName = () => {}` і не використовуйте ключове слово `function` (крім генераторів); якщо ви можете використовувати .bind для зв’язування параметрів, уникайте визначення функцій
- Повторне використання коду: зверніть увагу на повторне використання, виділіть більше дрібних функцій і рішуче уникайте великої кількості подібних або скопійованих структур коду.
- Доступ до об’єкта: визначте пріоритет використання призначення деструктуризації для вилучення необхідних атрибутів, уникайте повторного внутрішнього використання крапок для доступу до глибинних і вкладених атрибутів і об’єднуйте повторювані необов’язкові ланцюгові оцінки
- Не використовуйте об’єкти для параметрів функції, не пишіть `{a,b,c}`, пишіть `a, b, c`; якщо є багато додаткових параметрів, використовуйте [[кількість елементів конфігурації, елементи конфігурації], [кількість елементів конфігурації, елементи конфігурації],..] У цій парадигмі елементи конфігурації визначаються числовими константами
- Для повернення з кількома значеннями використовуйте масив `[a,b,c]` замість `{a,b,c}`. Якщо є більше 3 повернених значень, використовуйте числові константи, щоб визначити позиційне значення.
- Немає необхідності в рядку для представлення стану, використовуйте числові константи для визначення стану
- Шаблон рядка (``) не використовується, використовується конкатенація рядка (+), за винятком імпорту (зручно для статичного аналізу Vite)
- Цикл for, якщо вам потрібен серійний номер, використовуйте `++i` замість `i++`
- Уникайте непотрібного`?.`і забороняйте захисне програмування
- Ім’я змінної не обов’язково має бути у множині та закінчується на `_li`, щоб вказати список.

## Правила іменування

- Неймінг дотримується мінімалізму. Використовуйте короткі, але змістовні імена, наприклад: `rm` замість `remove`, `delete`, `del`. Однак вам також слід уникати крайнощів, наприклад: не використовуйте жодної літери `m` замість `map`
- Намагайтеся використовувати лише дієслова в іменуванні функцій. Якщо ви можете використовувати одне слово, щоб висловити це, не використовуйте два слова. Іменники відображаються в іменах файлів. Якщо вам потрібно додати дієслово до назви файлу, поставте іменник спочатку, а дієслово – останнє. Наприклад: `profileSet.js` замість `setProfile.js`
- Назва змінної: використовуйте стиль підкреслення (snake_case), наприклад `user_auth_token`; якщо змінна є функцією, використовуйте стиль імен у нижньому регістрі
- Назва функції: Використовуйте нижній регістр camelCase (camelCase), наприклад `userData`
- Параметри функції: якщо це функція зворотного виклику, назвіть її малим регістром, наприклад `onChange`
- Визначення констант на рівні модуля використовують стиль підкреслення у верхньому регістрі `UPPER_SNAKE_CASE`
- Не пишіть `get`, такі безглузді префікси, як наприклад: пишіть `cookieByHeader` замість `getCookie`
- Глобальні константи/константи на рівні модуля: використовуйте стиль підкреслення у верхньому регістрі (UPPER_SNAKE_CASE), напр. `CODE_TO_ID`, `ID_TO_LANG`

## Модульний механізм

- Імпорт: імпорт функціонує точно на вимогу, забороняє прямий імпорт усього модуля (уникайте `import * as x` або імпортуйте великі об’єкти)
- Експорт: вимкнути експорт об’єктів. Експортуйте функції та змінні як одиниці, наприклад `export const X=1, abc=()=>{};`. Спробуйте поєднати їх за допомогою const + кома, щоб оголосити експортований вміст. Якщо файл має лише одну функцію, використовуйте `export default`
- Якщо вам не потрібно внутрішньо викликати функцію експорту за замовчуванням, уникайте спочатку оголошувати константи, а потім експортувати їх у кінці файлу.
- Розділення шляху: під час отримання поточного шляху до каталогу необхідно використовувати `import.meta.dirname`

## помилка

- Уникайте рядкових помилок і використовуйте const для оголошення постійних кодів помилок.
- Якщо вам потрібно повернути неправильну інформацію про дані, використовуйте [код помилки, повідомлення про помилку,...]. Таким чином, повідомлення про помилку є не текстовим описом, а числовим значенням (наприклад, [FILE_OVERSIZE, file_size, max_size])
