# WebC.site

## Що це?

### Це рідна бібліотека веб-компонентів

Розроблений на основі [веб-компонентів](https://developer.mozilla.org/docs/Web/API/Web_components), він не залежить від середовища виконання та підходить для всіх фреймворків.

Надзвичайно оптимізований за розміром (одна десята розміру аналогічних популярних бібліотек).

Підтримує 75 мов. Ви можете безпосередньо імпортувати js і css для використання в Інтернеті або додавати компоненти до свого проекту, якщо це необхідно для додаткової розробки.

### Це структура розробки бібліотеки компонентів

Це статична версія блогу [storybook](https://storybook.js.org).

Створення бібліотеки компонентів може підвищити ефективність внутрішньої розробки компанії та створити імідж бренду в технічному колі.

На основі [webc.site](https://github.com/webc-site/webc.site) команди можуть легко створювати власні бібліотеки компонентів і публікувати їх на [сторінці github](https://pages.github.com), [сторінці cloudflare](https://pages.cloudflare.com) тощо. Опубліковані компоненти можна налагодити онлайн на [codepen.io](https://codepen.io) і [jsfiddle](https://jsfiddle.net).

Водночас вбудована структура перекладу може швидко реалізувати інтернаціоналізацію компонентів і документів, щоб аудиторія вашого продукту більше не була зв’язана мовою.

### Це нова парадигма front-end розробки для ери штучного інтелекту

Фреймворк вбудований `.agents/skills`, що дозволяє штучному інтелекту допомагати вам розробляти нові компоненти одним клацанням миші відповідно до найкращих практик.

<img src="https://i-01.eu.org/20260604121408079.avif" width="400">

Нехай інтерфейсний код зміниться з нежиттєздатної сміттєвої гори атмосферного програмування на стандартну частину, яку можна підтримувати та постійно розвивати протягом тривалого часу.

З огляду на характеристики розробки за допомогою штучного інтелекту, ми запропонували та розробили нову парадигму розробки інтерфейсу:

1. **Все є веб-компонентом**
2. **Компонент має бути незалежним від внутрішнього каналу передачі даних, а інтерфейс даних розкривається як функція зворотного виклику**

Таким чином штучний інтелект може генерувати підроблені дані та відображати різні стани компонентів на демонстраційній сторінці.

Розділення розробки компонентів та інтерфейсу даних у поєднанні з [платформою компонентів webc.site](https://github.com/webc-site/webc.site) Ще одна основна функція:

**Кожен компонент може запускати сервер розробки незалежно, розробляти та налагоджувати на демонстраційній сторінці**.

<img src="https://i-01.eu.org/webcSiteDemo.svg" width="400">

Це зменшує зовнішнє посилання для розробки та налагодження кожного компонента до `0`.

Таким чином, інструменти розробки за допомогою штучного інтелекту ([Claude Code](https://code.claude.com), [OpenAI Codex](https://openai.com), [Google Antigravity](https://antigravity.google), [Cursor](https://cursor.com), [Windsurf](https://windsurf.com), [Devin](https://devin.ai), [OpenHands](https://github.com/All-Hands-AI/OpenHands) тощо), ви можете насолоджуватися повністю автоматичною розробкою, відкритим налагодженням браузера та більше не блокуватись різні незручні фактори (дані серверної частини, статус входу, маршрутизація шляху тощо).

Якщо ви хочете контролювати штучний інтелект, зменшувати витрати та підвищувати ефективність, вам потрібна нова парадигма, яка мінімізує середовище розробки та налагодження кожного компонента.

#### Незалежний час виконання

Створена на основі стандартів [веб-компонентів](https://developer.mozilla.org/docs/Web/API/Web_components), бібліотека компонентів webc.site природно не залежить від часу виконання. Незалежно від того, використовуєте ви React, Vue, Angular, Svelte або традиційний чистий проект HTML/JS, усі компоненти можуть запускатися безпосередньо у браузері, як рідні HTML-теги, без необхідності вторинного пакування для конкретного фреймворку.

##### Порівняння з shadcn

[shadcn/ui](https://ui.shadcn.com) використовує режим розподілу коду, який вимагає копіювання коду компонента безпосередньо у ваш власний проект. Він тісно прив’язаний до екосистеми React і певних ланцюжків інструментів конструювання (таких як Tailwind CSS), і його не можна використовувати в інших фреймворках. Він також має певні вимоги до інженерної конфігурації проекту.

І компонент `webc.site`:
- **Універсальний кросфреймворк**: напишіть один раз, і його можна безпосередньо вводити та запускати в будь-якому зовнішньому фреймворку (навіть у середовищі без фреймворку).
- **Нульова залежність від середовища**: не потрібні спеціальні інструменти пакування чи препроцесори CSS, просто імпортуйте скомпільовані статичні файли, і їх можна використовувати відразу.

## Автоматизована розробка та налагодження

### Локальна розробка та налагодження

Кожен компонент веб-сторінки підтримує запуск сервера розробки незалежно, забезпечуючи мінімальне середовище розробки та налагодження:

- **Створіть новий компонент**: запустіть `./sh/new.js [css|js] [назва компонента]`, щоб швидко створити скелет розробки компонента (включаючи демонстраційну сторінку та конфігурацію інтернаціоналізації).
- **Незалежне налагодження**: запустіть `./dev.js [назва компонента]` (або запустіть `./dev.js` безпосередньо), щоб запустити сервер розробки Vite і автоматично відкрити демонстраційну сторінку компонента в браузері. Підтримує гаряче оновлення компонентів (HMR). Оскільки розробка та зв’язок із даними роз’єднані, існує нульове зовнішнє посилання для розробки, яке дуже підходить для штучного інтелекту (наприклад, Claude Code, Google Antigravity тощо), щоб допомогти в повністю автоматичній розробці та налагодженні браузера.

### Опублікувати бібліотеку компонентів

#### webc dist

Процес пакування та публікації високоавтоматизований:
- **Статичне пакування ресурсів**: запуск `./sh/dist.js` (тобто виконання завдання `webc dist`) може автоматично сканувати всі компоненти веб-сторінки під `src/`, компілювати, стискати та виводити їх у каталог `dist/`. Цей каталог містить файли JS/CSS для розповсюдження, демо-сайт, готовий для розгортання на сторінках GitHub або Cloudflare, і автоматично створені файли маршрутизації `_redirects` і `404.html`.
- **Публікація NPM**: запустіть `./sh/npmDist.js`, сценарій автоматично обробить залежності компонентів, збільшить номер версії пакета та виконає `npm publish` одним клацанням миші після завершення пакування, щоб опублікувати бібліотеку компонентів у NPM.

### За переклад відповідає платформа

Фреймворк має вбудовані автоматизовані процеси перекладу:
1. Налаштуйте правила перекладу (наприклад, вихідну мову, цільову мову та каталог для перекладу) у `tran.yml` кореневого каталогу проекту.
2. Використовуйте заповнювачі в компонентах або документах без жорсткого кодування кількох мов у своєму коді.
3. Запустіть `./sh/tran.js`, і механізм перекладу автоматично витягне текст і викличе службу перекладу WebC для обробки, автоматично генеруючи записи перекладу YAML і документацію README різними мовами.

### змінні середовища

Виконуючи автоматичний переклад, вам потрібно налаштувати такі змінні середовища, щоб авторизувати доступ до API перекладу:

- **`WEBC_SITE_TOKEN`** (або **`WEBC_TOKEN`**): ключ автентифікації, який використовується для доступу до API автоматичного перекладу `webc.site`. Можна ввести як змінну середовища або записати `token` у файл конфігурації `~/.config/webc.site.yml` у домашньому каталозі локального користувача.
