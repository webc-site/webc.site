# WebC.site

## 이건 뭐죠?

### 이것은 기본 웹 구성 요소 라이브러리입니다.

[웹 컴포넌트](https://developer.mozilla.org/docs/Web/API/Web_components)를 기반으로 개발되었으며 런타임 독립적이며 모든 프레임워크에 적합합니다.

크기가 매우 최적화되어 있습니다(유사한 인기 라이브러리 크기의 10분의 1 크기).

75개 언어를 지원합니다. 온라인 사용을 위해 js 및 css를 직접 가져오거나 보조 개발에 필요에 따라 프로젝트에 구성 요소를 추가할 수 있습니다.

<img src="https://i-01.eu.org/20260604114250265.avif" width="400">

### 이것이 컴포넌트 라이브러리 개발 프레임워크입니다.

[스토리북](https://storybook.js.org)의 정적 블로그 버전입니다.

구성 요소 라이브러리를 구축하면 회사 내부 개발 효율성이 향상되고 기술계에서 브랜드 이미지를 구축할 수 있습니다.

[webc.site](https://github.com/webc-site/webc.site)를 기반으로 팀은 쉽게 자체 구성 요소 라이브러리를 만들고 [github 페이지](https://pages.github.com), [cloudflare 페이지](https://pages.cloudflare.com) 등에 게시할 수 있습니다. 게시된 구성 요소는 [codepen.io](https://codepen.io) 및 [jsfiddle](https://jsfiddle.net)에서 온라인으로 디버깅할 수 있습니다.

동시에 내장된 번역 프레임워크는 구성 요소와 문서의 국제화를 빠르게 실현할 수 있으므로 제품 고객이 더 이상 언어에 얽매이지 않게 됩니다.

### 인공지능시대 프론트엔드 개발의 새로운 패러다임입니다.

프레임워크가 내장되어 있어 인공 지능이 모범 사례에 따라 한 번의 클릭으로 새로운 구성 요소를 개발하는 데 도움을 줄 수 있습니다.

<img src="https://i-01.eu.org/20260604121408079.avif" width="400">

프런트 엔드 코드를 지속 불가능한 대기 프로그래밍의 쓰레기 산에서 장기적으로 유지 관리하고 반복적으로 발전할 수 있는 표준 부분으로 변경하도록 하세요.

인공지능 기반 개발의 특성을 고려하여 새로운 프론트엔드 개발 패러다임을 제안하고 설계했습니다.

1. **모든 것이 웹 구성 요소입니다**
2. **구성요소는 백엔드 데이터 링크와 독립적이어야 하며 데이터 인터페이스는 콜백 함수로 노출됩니다**

이러한 방식으로 인공 지능은 가짜 데이터를 생성하고 데모 페이지에 구성 요소의 다양한 상태를 표시할 수 있습니다.

컴포넌트 개발과 데이터 인터페이스의 분리, [webc.site 컴포넌트 프레임워크](https://github.com/webc-site/webc.site)와 결합 또 다른 주요 기능:

**각 구성 요소는 개발 서버를 독립적으로 시작하고 데모 페이지에서 개발 및 디버그할 수 있습니다**.

<img src="https://i-01.eu.org/webcSiteDemo.svg" width="400">
<img src="https://i-01.eu.org/20260604112614499.avif" width="400">

이렇게 하면 각 구성 요소의 개발 및 디버깅을 위한 프런트 엔드 링크가 `0`로 줄어듭니다.

이를 통해 인공지능 기반 개발 도구([Claude Code](https://code.claude.com), [OpenAI Codex](https://openai.com), [Google Antigravity](https://antigravity.google), [Cursor](https://cursor.com), [Windsurf](https://windsurf.com), [Devin](https://devin.ai), [OpenHands](https://github.com/All-Hands-AI/OpenHands) 등)을 이용하면 완전 자동 개발, 오픈 브라우저 디버깅을 즐길 수 있으며, 더 이상 다양한 지저분한 요소(백엔드 데이터, 로그인)에 의해 차단되지 않습니다. 상태, 경로 라우팅 등).

인공지능을 제어하고 비용을 절감하며 효율성을 높이려면 각 구성 요소의 개발 및 디버깅 환경을 최소화하는 새로운 패러다임이 필요합니다.

### 웹 구성 요소 라이브러리

#### 궁극적인 크기 최적화

`webc.site`의 가상 스크롤 막대 구성요소는 `OverlayScrollbars`의 `10%`뿐입니다.

#### 런타임 독립적

표준 [웹 구성 요소](https://developer.mozilla.org/docs/Web/API/Web_components)을 기반으로 구축된 webc.site 구성 요소 라이브러리는 기본적으로 런타임에 구애받지 않습니다. React, Vue, Angular, Svelte 또는 전통적인 순수 HTML/JS 프로젝트를 사용하든 모든 구성 요소는 특정 프레임워크에 대한 보조 패키징 없이 기본 HTML 태그처럼 브라우저에서 직접 실행할 수 있습니다.

##### shadcn과의 비교

[shadcn/ui](https://ui.shadcn.com)에서는 구성 요소 코드를 자신의 프로젝트에 직접 복사해야 하는 코드 배포 모드를 채택합니다. 이는 React 생태계 및 특정 구성 도구 체인(예: Tailwind CSS)에 밀접하게 묶여 있으며 프레임워크 전체에서 사용할 수 없습니다. 또한 프로젝트 엔지니어링 구성에 대한 특정 요구 사항도 있습니다.

`webc.site`의 구성요소:
- **크로스 프레임워크 범용**: 한 번만 작성하면 모든 프런트엔드 프레임워크(프레임워크가 없는 환경에서도)에서 직접 도입하고 실행할 수 있습니다.
- **종속성이 없는 환경**: 특정 패키징 도구나 CSS 전처리기가 필요하지 않습니다. 컴파일된 정적 ​​파일을 가져오기만 하면 바로 사용할 수 있습니다.

## 자동화된 개발 및 디버깅

### 로컬 개발 및 디버깅

각 웹 페이지 구성 요소는 독립적으로 개발 서버 시작을 지원하여 최소한의 개발 및 디버깅 환경을 제공합니다.

- **새 구성 요소 만들기**: `./sh/new.js [css|js] [구성 요소 이름]`을 실행하여 구성 요소 개발 뼈대(데모 페이지 및 국제화 구성 포함)를 빠르게 생성합니다.
- **독립적 디버깅**: `./dev.js [구성요소 이름]`을 실행(또는 `./dev.js` 직접 실행)하여 Vite 개발 서버를 시작하고 브라우저에서 구성요소의 데모 페이지를 자동으로 엽니다. 구성 요소 핫 업데이트(HMR)를 지원합니다. 개발 및 데이터 링크가 분리되어 있기 때문에 개발 프런트엔드 링크가 없으며, 이는 완전 자동 개발 및 브라우저 디버깅을 지원하는 AI(예: Claude Code, Google Antigravity 등)에 매우 적합합니다.

### 구성 요소 라이브러리 게시

#### webc dist

패키징 및 게시 프로세스는 고도로 자동화되어 있습니다.
- **정적 리소스 패키징**: `./sh/dist.js` 실행(즉, `webc dist` 작업 실행)은 `src/` 아래의 모든 웹 페이지 구성 요소를 자동으로 검색하고 컴파일하고 압축하여 `dist/` 디렉터리에 출력할 수 있습니다. 이 디렉터리에는 프로덕션 배포용 JS/CSS 파일, GitHub 페이지 또는 Cloudflare 페이지에 배포할 준비가 된 데모 사이트, 자동으로 생성된 `_redirects` 및 `404.html` 라우팅 파일이 포함되어 있습니다.
- **NPM 게시**: `./sh/npmDist.js`을 실행하면 스크립트가 자동으로 구성 요소 종속성을 처리하고, 패키지 버전 번호를 높이고, 패키징이 완료된 후 한 번의 클릭으로 `npm publish`를 실행하여 구성 요소 라이브러리를 NPM에 게시합니다.

### 플랫폼은 번역을 담당합니다

프레임워크에는 자동 번역 프로세스가 내장되어 있습니다.
1. 프로젝트 루트 디렉터리의 `tran.yml`에서 번역 규칙(예: 원어, 대상 언어, 번역할 디렉터리)을 구성합니다.
2. 코드에 여러 언어를 하드코딩할 필요 없이 구성 요소나 문서에 자리 표시자를 사용할 수 있습니다.
3. `./sh/tran.js`을 실행하면 번역 엔진이 자동으로 텍스트를 추출하고 처리를 위해 WebC 번역 서비스를 호출하여 다양한 언어로 된 YAML 번역 항목과 README 문서를 자동으로 생성합니다.

### 환경 변수

자동 번역을 수행할 때 번역 API에 대한 액세스 권한을 부여하려면 다음 환경 변수를 구성해야 합니다.

- **`WEBC_SITE_TOKEN`** (또는 **`WEBC_TOKEN`**): `webc.site` 자동 번역 API에 액세스하는 데 사용되는 인증 키입니다. 환경 변수로 삽입하거나 로컬 사용자 홈 디렉터리의 `~/.config/webc.site.yml` 구성 파일에 `token` 작성할 수 있습니다.
