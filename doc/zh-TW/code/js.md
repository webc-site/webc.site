# JS 程式碼規範

## 程式碼風格

- 簡潔、優雅、高效，用最現代的 js 文法
- 拆分過長的函數，介面設計要低耦合、高內聚
- 多用陣列+循環 / forEach / map，少寫重複程式碼
- 簡單程式碼不寫函數註釋，避免過度註釋。複雜邏輯、特殊需求，用精煉的中文註釋
- 重複程式碼抽象化為函數，降低冗餘，例如 `cosnt A=new Table({ style: { border: NO_BORDER } }), B=new Table({ style: { border: NO_BORDER } })`，可以寫成 `const newTable=()=>new Table({ style: { border: NO_BORDER } }), A=newTable(), B=newTable()`
- 變數宣告：合併多個連續的 `const` 宣告為一個語句。要寫 `const a=1, b=2, c=3;`，而不是分三行寫
- 非同步處理：用 `await`，禁止使用 `.then` 鍊式調用
- 不要自動產生處理異常碼，不自動寫 `try...catch`（`try ... catch` 由人工維護，已有的 `try catch` 不要刪除）
- 純函數優先：只寫純函數，絕對不要寫 class
- 用箭頭函數 `const funcName = () => {}`，不使用 `function` 關鍵字(生成器除外)；如可用 .bind 綁定參數，就避免定義函數
- 程式碼重複使用：注重複用，多提取小函數，堅決避免出現大量類似或複製貼上的程式碼結構
- 物件存取：優先使用解構賦值提取所需的屬性，避免內部重複使用點號存取深層和嵌套屬性，並合併重複的可選鏈判斷
- 函數參數不要用對象，不寫 `{a,b,c}`，寫 `a, b, c`；如可選參數多，用[[配置項的數字,配置項],[配置項的數字,配置項],..] 這種範式，配置項用數字常數定義
- 多值返回，用數組 `[a,b,c]`，不是 `{a,b,c}`，如返回值超過3個，用數字常數定義位置含義
- 不用要字串表示狀態，用數字常數定義狀態
- 不用字串模板(``)，用字串拼接(+)，import 導入除外（方便 vite 靜態分析）
- for 迴圈，如需序號，用 `++i` 而非 `i++`
- 避免不必要的`?.`，禁止防禦式編程
- 變數名稱不用複數，用 `_li` 結尾，表示是列表

## 命名規範

- 命名追求極簡。使用盡量短但有意義的命名，例如：用 `rm` 而不是 `remove`、`delete`、`del`。但，也要避免走極端，例如：不要用單字母 `m` 取代 `map`
- 函數命名盡量只用動詞，可以用一個單字表達，就不要用兩個單字。名詞用檔案命名體現，如果有需要在檔案名稱中加入動詞，請名詞在前，動詞在後。例如：是 `profileSet.js` 而不是 `setProfile.js`
- 變數名稱: 使用底線風格 (snake_case)，例如 `user_auth_token`
- 函數名稱: 使用小寫駝峰風格 (camelCase)，例如 `userData`
- 函數參數: 如是回呼函數，用小寫駝峰命名，如 `onChange`
- 模組級常數定義用大寫下劃線風格 `UPPER_SNAKE_CASE`
- 不寫 `get` 這類沒意義前綴，如: 寫 `cookieByHeader`，而不是 `getCookie`
- 全域/模組級常數：使用大寫下劃線風格 (UPPER_SNAKE_CASE)，例如 `CODE_TO_ID`、`ID_TO_LANG`

## 模組化機制

- 導入：按需精準導入函數，禁止直接導入整個模組（避免 `import * as x` 或導入大物件）
- 導出：禁止導出對象。以函數、變數為單元導出，例如 `export const X=1, abc=()=>{};`,盡量合併用一個 const + 逗號來宣告導出的內容,如果一個檔案只有一個函數，用 `export default`
- 除非需要內部呼叫 export default 的函數，否則避免先宣告常數再在檔案末端匯出
- 路徑解析：取得目前目錄路徑時，必須使用 `import.meta.dirname`

## 錯誤

- 避免用字串錯誤，盡量用 const 宣告 常數錯誤碼
- 如需要傳回錯誤的資料訊息，用 [錯誤碼,錯誤訊息,...] 這種方式，錯誤訊息不是文字描述，而是數值之類的(例如[FILE_OVERSIZE, file_size, max_size])

- 加解密：強制使用原生的 Web Crypto API
- 二進位資料：處理二進位時，盡量統一使用 `Uint8Array`
