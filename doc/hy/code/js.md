# JS կոդի ճշգրտում

## կոդավորման ոճը

- Պարզ, էլեգանտ և արդյունավետ՝ օգտագործելով ամենաժամանակակից js շարահյուսությունը
- Բաժանեք չափազանց երկար գործառույթները, և ինտերֆեյսի դիզայնը պետք է լինի ցածր միացում և բարձր համախմբվածություն:
- Ավելի շատ օգտագործեք զանգվածներ + հանգույցներ/forEach/քարտեզ և գրեք ավելի քիչ կրկնվող կոդ
- Մի գրեք գործառույթի մեկնաբանությունները պարզ կոդով՝ ավելորդ մեկնաբանություններից խուսափելու համար: Բարդ տրամաբանությունը և հատուկ կարիքները բացատրվում են նուրբ չինարենով:
- Կրկնվող կոդը վերացվում է ֆունկցիաների մեջ՝ ավելորդությունը նվազեցնելու համար: Օրինակ, `cosnt A=new Table({ style: { border: NO_BORDER } }), B=new Table({ style: { border: NO_BORDER } })` կարելի է գրել `const newTable=()=>new Table({ style: { border: NO_BORDER } }), A=newTable(), B=newTable()`
- Փոփոխական հռչակագիր. Միավորել մի քանի հաջորդական `const` հայտարարություններ մեկ հայտարարության մեջ: Երեք տողով գրելու փոխարեն գրեք `const a=1, b=2, c=3;`
- Ասինխրոն մշակում. օգտագործեք `await`, արգելեք `.then` շղթայական զանգերի օգտագործումը
- Ավտոմատ մի ստեղծեք բացառությունների մշակման կոդ, ինքնաբերաբար մի գրեք `try...catch` (`try ... catch` պահպանվում է ձեռքով, գոյություն ունեցող `try catch`-ը չպետք է ջնջվի)
- Մաքուր ֆունկցիաներ նախ. գրել միայն մաքուր ֆունկցիաներ, երբեք դասեր
- Օգտագործեք սլաքների գործառույթները `const funcName = () => {}` և մի օգտագործեք `function` հիմնաբառը (բացի գեներատորներից); եթե դուք կարող եք օգտագործել .bind պարամետրերը կապելու համար, խուսափեք գործառույթներ սահմանելուց
- Կոդի կրկնակի օգտագործում. Ուշադրություն դարձրեք կրկնակի օգտագործմանը, հանեք ավելի փոքր գործառույթներ և վճռականորեն խուսափեք մեծ թվով նմանատիպ կամ պատճենահանված կոդերի կառուցվածքներից:
- Օբյեկտի հասանելիություն. առաջնահերթություն տվեք օգտագործել ապակառուցողական հանձնարարություն՝ պահանջվող ատրիբուտները հանելու համար, խուսափեք ներսում կրկնվող կետերից՝ խորը և ներկառուցված ատրիբուտներին մուտք գործելու համար և միաձուլեք կրկնվող կամընտիր շղթայի դատողությունները:
- Մի օգտագործեք օբյեկտներ ֆունկցիայի պարամետրերի համար, մի գրեք `{a,b,c}`, գրեք `a, b, c`; եթե կան բազմաթիվ ընտրովի պարամետրեր, օգտագործեք [[կազմաձեւման տարրերի թիվը, կազմաձեւման տարրերը], [կազմաձեւման տարրերի թիվը, կազմաձեւման տարրերը],...] Այս պարադիգմում կազմաձևման տարրերը սահմանվում են թվային հաստատուններով։
- Բազմարժեք վերադարձի համար օգտագործեք `[a,b,c]` զանգվածը՝ `{a,b,c}`-ի փոխարեն: Եթե ​​կան 3-ից ավելի վերադարձված արժեքներ, օգտագործեք թվային հաստատուններ՝ դիրքային նշանակությունը սահմանելու համար:
- Վիճակը ներկայացնելու համար տողի կարիք չկա, վիճակը սահմանելու համար օգտագործեք թվային հաստատուններ
- Ոչ մի տողերի ձևանմուշ (``) չի օգտագործվում, օգտագործվում է տողերի միացում (+), բացառությամբ ներմուծման (հարմար է Vite ստատիկ վերլուծության համար)
- Օղակի համար, եթե ձեզ անհրաժեշտ է սերիական համար, օգտագործեք `++i`՝ `i++`-ի փոխարեն:
- Խուսափեք ավելորդ`?.`ից և արգելեք պաշտպանական ծրագրավորումը
- Փոփոխականի անունը պետք չէ հոգնակի լինել և վերջանում է `_li`-ով` ցանկը նշելու համար:

## Անվանման կոնվենցիա

- Անվանումը մինիմալիզմ է հետապնդում։ Օգտագործեք կարճ, բայց իմաստալից անուններ, օրինակ՝ օգտագործեք `rm`՝ `remove`, `delete`, `del` փոխարեն: Այնուամենայնիվ, դուք նույնպես պետք է խուսափեք ծայրահեղությունների գնալուց, օրինակ՝ մի օգտագործեք ոչ մի տառ `m` `map`-ի փոխարեն:
- Գործառույթների անվանման մեջ փորձեք օգտագործել միայն բայեր: Եթե ​​դուք կարող եք օգտագործել մեկ բառ այն արտահայտելու համար, մի օգտագործեք երկու բառ: Գոյականներն արտացոլվում են ֆայլերի անուններում: Եթե ​​Ձեզ անհրաժեշտ է ֆայլի անվան մեջ բայ ավելացնել, խնդրում ենք սկզբում դնել գոյականը, իսկ վերջում՝ բայը: Օրինակ՝ `profileSet.js` `setProfile.js`-ի փոխարեն
- Փոփոխականի անունը. օգտագործել ընդգծված ոճը (snake_case), օրինակ `user_auth_token`; եթե փոփոխականը ֆունկցիա է, օգտագործեք փոքրատառ ուղտի անվանման ոճը
- Գործառույթի անվանումը՝ օգտագործել փոքրատառ camelCase (camelCase), օրինակ `userData`
- Ֆունկցիայի պարամետրեր. Եթե դա հետադարձ զանգի ֆունկցիա է, անվանեք այն փոքրատառով, օրինակ՝ `onChange`
- Մոդուլի մակարդակի հաստատուն սահմանումները օգտագործում են մեծատառ ընդգծված ոճ `UPPER_SNAKE_CASE`
- Մի գրեք `get` այնպիսի անիմաստ նախածանցներ, ինչպիսիք են՝ `getCookie`-ի փոխարեն գրել `cookieByHeader`
- Համաշխարհային/մոդուլի մակարդակի հաստատուններ. օգտագործեք մեծատառ ընդգծված ոճ (UPPER_SNAKE_CASE), օրինակ. `CODE_TO_ID`, `ID_TO_LANG`

## Մոդուլային մեխանիզմ

- Ներմուծում. Ներմուծեք գործառույթները ճշգրիտ ըստ պահանջի, արգելեք ամբողջ մոդուլի ուղղակի ներմուծումը (խուսափեք `import * as x` կամ ներմուծեք մեծ օբյեկտներ)
- Արտահանում. անջատել օբյեկտների արտահանումը: Արտահանեք ֆունկցիաները և փոփոխականները որպես միավորներ, ինչպիսիք են `export const X=1, abc=()=>{};`: Փորձեք դրանք համատեղել const + ստորակետով՝ արտահանվող բովանդակությունը հայտարարագրելու համար: Եթե ​​ֆայլն ունի միայն մեկ գործառույթ, օգտագործեք `export default`
- Եթե ​​կարիք չկա արտահանման լռելյայն ֆունկցիան ներքին կարգով կանչելու, նախ խուսափեք հաստատուններ հայտարարելուց և այնուհետև դրանք ֆայլի վերջում արտահանելուց:
- Ուղու լուծում. Ընթացիկ գրացուցակի ուղին ստանալիս պետք է օգտագործվի `import.meta.dirname`

## սխալ

- Խուսափեք լարային սխալներից և օգտագործեք const՝ հաստատուն սխալի կոդեր հայտարարելու համար:
- Եթե ​​Ձեզ անհրաժեշտ է սխալ տվյալների տեղեկատվությունը վերադարձնել, օգտագործեք [error code, error message,...] Այս կերպ սխալի հաղորդագրությունը ոչ թե տեքստային նկարագրություն է, այլ թվային արժեք (օրինակ՝ [FILE_OVERSIZE, file_size, max_size])
