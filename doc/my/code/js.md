# JS ကုဒ်သတ်မှတ်ချက်

## coding ပုံစံ

- ခေတ်အမီဆုံး js syntax ကို အသုံးပြု၍ ရိုးရှင်း၊ ပြေပြစ်ပြီး ထိရောက်မှုရှိသည်။
- ရှည်လျားလွန်းသော လုပ်ဆောင်ချက်များကို ခွဲထုတ်ပြီး အင်တာဖေ့စ် ဒီဇိုင်းသည် အချိတ်အဆက် နည်းပါးပြီး ပေါင်းစည်းမှု မြင့်မားသင့်သည်။
- arrays + loops/forEach/map ကို ပိုမိုအသုံးပြုပြီး ထပ်ကျော့ကုဒ်ကို လျှော့ရေးပါ။
- အလွန်အကျွံ မှတ်ချက်များကို ရှောင်ရှားရန် ရိုးရှင်းသော ကုဒ်တွင် လုပ်ဆောင်ချက် မှတ်ချက်များ မရေးပါနှင့်။ ရှုပ်ထွေးသောယုတ္တိဗေဒနှင့် အထူးလိုအပ်ချက်များကို သန့်စင်သော တရုတ်ဘာသာဖြင့် ရှင်းပြထားသည်။
- ထပ်ခါတလဲလဲ ကုဒ်ကို ရှုပ်ထွေးမှု လျှော့ချရန် လုပ်ဆောင်ချက်များတွင် ပါ၀င်သည်။ ဥပမာအားဖြင့်၊ `cosnt A=new Table({ style: { border: NO_BORDER } }), B=new Table({ style: { border: NO_BORDER } })`ကို `const newTable=()=>new Table({ style: { border: NO_BORDER } }), A=newTable(), B=newTable()` အဖြစ် ရေးသားနိုင်ပါသည်။
- ပြောင်းလဲနိုင်သော ကြေငြာချက်- များစွာသော ဆက်တိုက် `const` ကြေငြာချက်များကို ဖော်ပြချက်တစ်ခုသို့ ပေါင်းစပ်ပါ။ စာကြောင်းသုံးကြောင်းဖြင့်ရေးမည့်အစား `const a=1, b=2, c=3;` ဟုရေးပါ။
- Asynchronous လုပ်ဆောင်ခြင်း- `await`ကိုသုံးပါ၊ `.then` ကွင်းဆက်ခေါ်ဆိုမှုများကို အသုံးပြုခြင်းကို တားမြစ်ပါ
- ခြွင်းချက် ကိုင်တွယ်မှုကုဒ်ကို အလိုအလျောက် မထုတ်ပေးပါနှင့်၊ အလိုအလျောက် မရေးပါနှင့် `try...catch` (`try ... catch`ကို ကိုယ်တိုင် ထိန်းသိမ်းထားပြီး၊ ရှိပြီးသား `try catch`ကို မဖျက်သင့်ပါ)
- သန့်ရှင်းသောလုပ်ဆောင်ချက်များကို ဦးစွာပထမ- သန့်စင်သောလုပ်ဆောင်ချက်များကိုသာရေးပါ၊ အတန်းများကိုဘယ်တော့မှမရေးပါ။
- မြှားလုပ်ဆောင်ချက်များကို `const funcName = () => {}`အသုံးပြုပြီး `function`သော့ချက်စကားလုံး (မီးစက်မှလွဲ၍) မသုံးပါနှင့်။ ဘောင်များကို စည်းရန် .bind ကို သုံးနိုင်လျှင် လုပ်ဆောင်ချက်များကို အဓိပ္ပါယ်ဖွင့်ဆိုခြင်းကို ရှောင်ကြဉ်ပါ။
- ကုဒ်ကို ပြန်လည်အသုံးပြုခြင်း- ပြန်လည်အသုံးပြုရန်၊ ပိုမိုသေးငယ်သော လုပ်ဆောင်ချက်များကို ထုတ်ယူရန် အာရုံစိုက်ပြီး အလားတူ သို့မဟုတ် မိတ္တူကူးထည့်ထားသော ကုဒ်ဖွဲ့စည်းပုံအများအပြားကို ပြတ်သားစွာ ရှောင်ကြဉ်ပါ။
- အရာဝတ္တုဝင်ရောက်ခွင့်- လိုအပ်သော အရည်အချင်းများကို ထုတ်ယူရန်၊ နက်ရှိုင်းသော နှင့် အသိုက်အမြုံထားသော အင်္ဂါရပ်များကို ဝင်ရောက်ကြည့်ရှုရန် ထပ်ခါတလဲလဲ အသုံးပြုခြင်းကို ရှောင်ကြဉ်ပြီး ထပ်ခါတလဲလဲ ရွေးချယ်နိုင်သော ကွင်းဆက်အကဲဖြတ်ချက်များကို ပေါင်းစည်းရန်၊
- လုပ်ဆောင်ချက်ဘောင်များအတွက် အရာဝတ္တုများကို အသုံးမပြုပါနှင့်၊ `{a,b,c}`ကို မရေးပါနှင့်၊ `a, b, c`ဟု မရေးပါနှင့်။ ရွေးချယ်နိုင်သော ကန့်သတ်ဘောင်များစွာရှိပါက၊ [[ဖွဲ့စည်းမှုဆိုင်ရာအရာများ၊ ဖွဲ့စည်းမှုဆိုင်ရာအရာများ]၊ [ဖွဲ့စည်းမှုဆိုင်ရာအရာများ အရေအတွက်၊ ပုံစံဖွဲ့စည်းမှုဆိုင်ရာအရာများ]၊..] ဤပါရာဒိုင်းတွင်၊ ဖွဲ့စည်းမှုဆိုင်ရာအရာများကို ကိန်းသေကိန်းသေများဖြင့် သတ်မှတ်ပေးသည်။
- တန်ဖိုးများစွာပြန်ပို့ရန်အတွက် `{a,b,c}`အစား array `[a,b,c]`ကို သုံးပါ။ ပြန်ပေးသည့်တန်ဖိုး 3 ခုထက်ပိုပါက၊ positional အဓိပ္ပာယ်ကိုသတ်မှတ်ရန် ဂဏန်းကိန်းသေများကို အသုံးပြုပါ။
- ပြည်နယ်ကို ကိုယ်စားပြုရန် စာကြောင်းတစ်ကြောင်း မလိုအပ်ပါ၊ ပြည်နယ်ကို သတ်မှတ်ရန် ကိန်းသေများကို အသုံးပြုပါ။
- string template (``) ကို အသုံးမပြုပါ၊ တင်သွင်းခြင်းမှလွဲ၍ string concatenation (+) ကို အသုံးပြုပါသည် (Vite static analysis အတွက် အဆင်ပြေသည်)
- ကွင်းဆက်အတွက်၊ သင်သည် အမှတ်စဉ်နံပါတ်တစ်ခု လိုအပ်ပါက၊ `i++`အစား `++i`ကို သုံးပါ။
- မလိုအပ်သော`?.`ကို ရှောင်ကြဉ်ပြီး ခုခံကာကွယ်မှု ပရိုဂရမ်များကို တားမြစ်ပါ။
- ပြောင်းလဲနိုင်သောအမည်သည် အများကိန်းဖြစ်ရန်မလိုအပ်ပါ၊ စာရင်းတစ်ခုကိုညွှန်ပြရန် `_li` နှင့်အဆုံးသတ်သည်။

## အမည်ပေးခြင်း သဘောတူညီချက်

- အမည်ပေးခြင်းသည် minimalism ကိုလိုက်စားသည်။ တိုတောင်းသော်လည်း အဓိပ္ပာယ်ရှိသော အမည်များကို သုံးပါ- ဥပမာ- `remove`၊ `delete`၊ `del`အစား `rm`ကို သုံးပါ။ သို့သော်၊ သင်သည် လွန်ကဲသွားခြင်းကို ရှောင်ကြဉ်သင့်သည်၊ ဥပမာ- `m`စာလုံးတစ်လုံးတည်းကို `map`အစား မသုံးပါနှင့်။
- လုပ်ဆောင်ချက်အမည်ပေးခြင်းတွင် ကြိယာများကိုသာ အသုံးပြုရန် ကြိုးစားပါ။ ၎င်းကိုဖော်ပြရန် စကားလုံးတစ်လုံးသုံးနိုင်လျှင် စကားလုံးနှစ်လုံးမသုံးပါနှင့်။ နာမ်များသည် ဖိုင်အမည်များတွင် ထင်ဟပ်နေသည်။ ဖိုင်အမည်တွင် ကြိယာတစ်ခုထည့်ရန် လိုအပ်ပါက ကျေးဇူးပြု၍ နာမ်ကို ဦးစွာပထမနှင့် ကြိယာကို နောက်ဆုံးထားပေးပါ။ ဥပမာ- `profileSet.js` `setProfile.js` အစား
- ပြောင်းလဲနိုင်သောအမည်များ- ဥပမာအားဖြင့် `user_auth_token` အောက်အမှတ်ပုံစံ (snake_case) ကိုသုံးပါ။
- လုပ်ဆောင်ချက်အမည်- ဥပမာ `userData` ကုလားအုတ်အသေး (camelCase) ကို အသုံးပြုပါ။
- လုပ်ဆောင်ချက် ကန့်သတ်ချက်များ- ၎င်းသည် ပြန်ခေါ်သည့် လုပ်ဆောင်ချက်ဖြစ်ပါက၊ ၎င်းကို `onChange`ကဲ့သို့ ကုလားအုတ်အသေးဖြင့် အမည်ပေးခြင်း၊
- မော်ဂျူးအဆင့် အဆက်မပြတ် အဓိပ္ပါယ်ဖွင့်ဆိုချက်များသည် စာလုံးကြီးအောက်ခံပုံစံ `UPPER_SNAKE_CASE`ကို အသုံးပြုသည်။
- `get`၊ ကဲ့သို့သော အဓိပ္ပါယ်မဲ့ ရှေ့ဆက်များကို `cookieByHeader` `getCookie`အစား မရေးပါနှင့်။
- Global/module-level constants- စာလုံးကြီး မျဉ်းကြောင်းပုံစံ (UPPER_SNAKE_CASE) ကိုသုံးပါ၊ ဥပမာ။ `CODE_TO_ID`၊ `ID_TO_LANG`

## Modular ယန္တရား

- တင်သွင်းခြင်း- လိုအပ်ချက်အရ လုပ်ဆောင်ချက်များကို တိကျစွာတင်သွင်းပါ၊ မော်ဂျူးတစ်ခုလုံးကို တိုက်ရိုက်တင်သွင်းခြင်းကို တားမြစ်ပါ ( `import * as x` သို့မဟုတ် ကြီးမားသောအရာဝတ္ထုများကို တင်သွင်းခြင်းကို ရှောင်ကြဉ်ပါ)
- တင်ပို့ခြင်း- အရာဝတ္ထုများ တင်ပို့ခြင်းကို ပိတ်ပါ။ `export const X=1, abc=()=>{};`ကဲ့သို့သော ယူနစ်များအဖြစ် လုပ်ဆောင်ချက်များနှင့် ကိန်းရှင်များကို ထုတ်ယူပါ။ တင်ပို့ထားသော အကြောင်းအရာကို ကြေညာရန် ၎င်းတို့ကို const + ကော်မာဖြင့် ပေါင်းစပ်ကြည့်ပါ။ ဖိုင်တစ်ခုတွင် လုပ်ဆောင်ချက်တစ်ခုသာရှိလျှင် `export default`ကို အသုံးပြုပါ။
- တင်ပို့မှု ပုံသေလုပ်ဆောင်ချက်ကို အတွင်းပိုင်း၌ သင်ခေါ်ဆိုရန်မလိုအပ်ပါက၊ ကိန်းသေများကို ဦးစွာကြေညာပြီးနောက် ဖိုင်၏အဆုံးတွင် ၎င်းတို့ကို ထုတ်ယူခြင်းကို ရှောင်ကြဉ်ပါ။
- လမ်းကြောင်းဖြေရှင်းချက်- လက်ရှိလမ်းညွှန်လမ်းကြောင်းကို ရယူသည့်အခါ၊ `import.meta.dirname`ကို အသုံးပြုရပါမည်။

## အမှား

- string အမှားများကိုရှောင်ပြီး စဉ်ဆက်မပြတ်အမှားအယွင်းကုဒ်များကိုကြေငြာရန် const ကိုသုံးပါ။
- မှားယွင်းသောဒေတာအချက်အလက်များကို ပြန်ပေးရန် လိုအပ်ပါက၊ [အမှားကုဒ်၊ အမှားအယွင်းမက်ဆေ့ချ်၊...] ကိုသုံးပါ ဤနည်းအားဖြင့် အမှားမက်ဆေ့ချ်သည် စာသားဖော်ပြချက်မဟုတ်သော်လည်း ဂဏန်းတန်ဖိုး (ဥပမာ [FILE_OVERSIZE၊ ဖိုင်_အရွယ်အစား၊ max_size])၊
