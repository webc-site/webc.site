# מפרט קוד JS

## סגנון קידוד

- פשוט, אלגנטי ויעיל, תוך שימוש בתחביר js המודרני ביותר
- פיצול פונקציות ארוכות מדי, ועיצוב הממשק צריך להיות צימוד נמוך ולכידות גבוהה.
- השתמש יותר במערכים + לולאות/לכל אחד/מפה וכתוב פחות קוד שחוזר על עצמו
- אל תכתוב הערות פונקציה בקוד פשוט כדי למנוע הערות מוגזמות. היגיון מורכב וצרכים מיוחדים מוסברים בסינית מעודנת.
- קוד חוזר מופשט לפונקציות כדי להפחית יתירות. לדוגמה, ניתן לכתוב `cosnt A=new Table({ style: { border: NO_BORDER } }), B=new Table({ style: { border: NO_BORDER } })` בתור `const newTable=()=>new Table({ style: { border: NO_BORDER } }), A=newTable(), B=newTable()`
- הצהרת משתנים: שלב מספר `const` הצהרות עוקבות להצהרה אחת. במקום לכתוב את זה בשלוש שורות, כתוב `const a=1, b=2, c=3;`
- עיבוד אסינכרוני: השתמש ב-`await`, אסור להשתמש ב`.then` שיחות שרשרת
- אל תיצור אוטומטית קוד טיפול בחריגים, אל תכתוב אוטומטית `try...catch` (`try ... catch` נשמר באופן ידני, אין למחוק את ה`try catch` הקיים)
- תחילה פונקציות טהורות: כתוב רק פונקציות טהורות, לעולם לא מחלקות
- השתמש בפונקציות החצים `const funcName = () => {}` ואל תשתמש במילת המפתח `function` (למעט מחוללים); אם אתה יכול להשתמש ב-.bind כדי לאגד פרמטרים, הימנע מהגדרת פונקציות
- שימוש חוזר בקוד: שימו לב לשימוש חוזר, חלץ עוד פונקציות קטנות והימנע בנחישות ממספר רב של מבני קוד דומים או מודבקים בהעתקה.
- גישה לאובייקט: תעדוף שימוש בהקצאת ביטול מבנה כדי לחלץ תכונות נדרשות, הימנע משימוש חוזר בנקודות פנימי כדי לגשת לתכונות עמוקות ומקוננות, ומיזוג שיפוטי שרשרת אופציונליים חוזרים ונשנים
- אל תשתמש באובייקטים לפרמטרים של פונקציה, אל תכתוב `{a,b,c}`, תכתוב `a, b, c`; אם ישנם פרמטרים אופציונליים רבים, השתמש ב-[[מספר פריטי תצורה, פריטי תצורה], [מספר פריטי תצורה, פריטי תצורה],..] בפרדיגמה זו, פריטי תצורה מוגדרים בקבועים מספריים
- עבור החזרות מרובות ערכים, השתמש במערך `[a,b,c]` במקום `{a,b,c}`. אם יש יותר מ-3 ערכי החזרה, השתמש בקבועים מספריים כדי להגדיר את המשמעות המיקוםית.
- אין צורך במחרוזת שתייצג את המצב, השתמש בקבועים מספריים כדי להגדיר את המצב
- לא נעשה שימוש בתבנית מחרוזת (``), נעשה שימוש בשרשור מחרוזת (+), למעט ייבוא ​​(נוח לניתוח סטטי של Vite)
- עבור לולאה, אם אתה צריך מספר סידורי, השתמש ב-`++i` במקום ב-`i++`
- הימנע מ`?.`מיותר ואסר על תכנות הגנתי
- שם המשתנה אינו צריך להיות רבים, ומסתיים ב`_li` כדי לציין רשימה.

## מוסכמות שמות

- השם רודף מינימליזם. השתמש בשמות קצרים אך בעלי משמעות, לדוגמה: השתמש ב-`rm` במקום ב-`remove`, `delete`, `del`. עם זאת, כדאי גם להימנע מלהגיע לקיצוניות, למשל: אל תשתמש באות בודדת `m` במקום `map`
- נסה להשתמש רק בפעלים במתן שמות לפונקציות. אם אתה יכול להשתמש במילה אחת כדי לבטא אותה, אל תשתמש בשתי מילים. שמות עצם משתקפים בשמות הקבצים. אם אתה צריך להוסיף פועל לשם הקובץ, נא לשים את שם העצם ראשון ואת הפועל אחרון. לדוגמה: `profileSet.js` במקום `setProfile.js`
- שם משתנה: השתמש בסגנון קו תחתון (סnake_case), לדוגמה `user_auth_token`; אם המשתנה הוא פונקציה, השתמש בסגנון מתן שמות באותיות קטנות של גמל
- שם הפונקציה: השתמש באותיות קטנות של camelCase (camelCase), לדוגמה `userData`
- פרמטרים של פונקציה: אם מדובר בפונקציית התקשרות חוזרת, שם אותה באותיות קטנות של גמל, כגון `onChange`
- הגדרות קבועות ברמת המודול משתמשות בסגנון קו תחתון רישיות `UPPER_SNAKE_CASE`
- אל תכתוב `get`, קידומות חסרות משמעות כגון: כתוב `cookieByHeader` במקום `getCookie`
- קבועים גלובליים/רמת מודול: השתמש בסגנון קו תחתון גדול (UPPER_SNAKE_CASE), למשל. `CODE_TO_ID`, `ID_TO_LANG`

## מנגנון מודולרי

- ייבוא: ייבא פונקציות בצורה מדויקת לפי דרישה, איסור ייבוא ​​ישיר של המודול כולו (הימנעו מ`import * as x` או ייבא אובייקטים גדולים)
- ייצוא: השבת ייצוא של אובייקטים. ייצא פונקציות ומשתנים כיחידות, כגון `export const X=1, abc=()=>{};`. נסה לשלב אותם עם const + פסיק כדי להצהיר על התוכן המיוצא. אם לקובץ יש רק פונקציה אחת, השתמש ב-`export default`
- אלא אם כן עליך לקרוא לפונקציית ברירת המחדל של הייצוא באופן פנימי, הימנע מהכרזת קבועים תחילה ולאחר מכן לייצא אותם בסוף הקובץ.
- רזולוציית נתיב: בעת קבלת נתיב הספרייה הנוכחי, יש להשתמש ב`import.meta.dirname`
