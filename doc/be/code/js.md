# Спецыфікацыя кода JS

## стыль кадавання

- Просты, элегантны і эфектыўны, з выкарыстаннем самага сучаснага сінтаксісу js
- Падзяліце празмерна доўгія функцыі, а дызайн інтэрфейсу павінен мець нізкі ўзровень сувязі і высокую згуртаванасць.
- Больш выкарыстоўвайце масівы + цыклы/forEach/map і пішыце менш паўтаральнага кода
- Не пішыце каментарыі да функцый у простым кодзе, каб пазбегнуць празмерных каментарыяў. Складаная логіка і асаблівыя патрэбы тлумачацца на вытанчанай кітайскай мове.
- Паўтаральны код абстрагуецца ў функцыі, каб паменшыць празмернасць. Напрыклад, `cosnt A=new Table({ style: { border: NO_BORDER } }), B=new Table({ style: { border: NO_BORDER } })` можна запісаць як `const newTable=()=>new Table({ style: { border: NO_BORDER } }), A=newTable(), B=newTable()`
- Дэкларацыя зменнай: аб'яднайце некалькі паслядоўных дэкларацый `const` у адну заяву. Замест запісу ў тры радкі напішыце `const a=1, b=2, c=3;`
- Асінхронная апрацоўка: выкарыстоўваць `await`, забараніць выкарыстанне `.then` ланцуговых выклікаў
- Не ствараць аўтаматычна код апрацоўкі выключэнняў, не пісаць аўтаматычна `try...catch` (`try ... catch` падтрымліваецца ўручную, існуючы `try catch` не павінен быць выдалены)
- Спачатку чыстыя функцыі: пішыце толькі чыстыя функцыі, а не класы
- Выкарыстоўвайце функцыі стрэлак `const funcName = () => {}` і не выкарыстоўвайце ключавое слова `function` (акрамя генератараў); калі вы можаце выкарыстоўваць .bind для прывязкі параметраў, пазбягайце вызначэння функцый
- Паўторнае выкарыстанне кода: звяртайце ўвагу на паўторнае выкарыстанне, здабывайце больш невялікіх функцый і рашуча пазбягайце вялікай колькасці падобных або скапіяваных структур кода.
- Доступ да аб'ектаў: ​​расстаўляйце прыярытэты пры прызначэнні дэструктурызацыі для вылучэння неабходных атрыбутаў, пазбягайце шматразовага ўнутранага выкарыстання кропак для доступу да глыбокіх і ўкладзеных атрыбутаў і аб'ядноўвайце паўторныя неабавязковыя ланцужныя меркаванні
- Не выкарыстоўвайце аб'екты для параметраў функцыі, не пішыце `{a,b,c}`, пішыце `a, b, c`; калі ёсць шмат неабавязковых параметраў, выкарыстоўвайце [[колькасць элементаў канфігурацыі, элементы канфігурацыі], [колькасць элементаў канфігурацыі, элементы канфігурацыі],..] У гэтай парадыгме элементы канфігурацыі вызначаюцца лікавымі канстантамі
- Для вяртання некалькіх значэнняў выкарыстоўвайце масіў `[a,b,c]` замест `{a,b,c}`. Калі вяртаецца больш за 3 значэнні, выкарыстоўвайце лікавыя канстанты для вызначэння пазіцыйнага значэння.
- Няма неабходнасці ў радку для прадстаўлення стану, выкарыстоўвайце лікавыя канстанты для вызначэння стану
- Шаблон радка (``) не выкарыстоўваецца, выкарыстоўваецца канкатэнацыя радка (+), за выключэннем імпарту (зручна для статычнага аналізу Vite)
- У цыкле for, калі вам патрэбны серыйны нумар, выкарыстоўвайце `++i` замест `i++`
- Пазбягайце непатрэбнага`?.`і забараняйце абарончае праграмаванне
- Імя зменнай не павінна быць у множным ліку і заканчваецца на `_li`, каб паказаць спіс.

## Правілы наймення

- Неймінг імкнецца да мінімалізму. Выкарыстоўвайце кароткія, але значныя імёны, напрыклад: выкарыстоўвайце `rm` замест `remove`, `delete`, `del`. Аднак вы таксама павінны пазбягаць крайнасці, напрыклад: не выкарыстоўвайце адну літару `m` замест `map`
- Старайцеся выкарыстоўваць толькі дзеясловы ў найменні функцый. Калі вы можаце выкарыстоўваць адно слова, каб выказаць гэта, не выкарыстоўвайце два словы. Назоўнікі адлюстроўваюцца ў назвах файлаў. Калі вам трэба дадаць дзеяслоў да назвы файла, пастаўце назоўнік першым, а дзеяслоў - апошнім. Напрыклад: `profileSet.js` замест `setProfile.js`
- Імёны зменных: выкарыстоўвайце стыль падкрэслівання (snake_case), напрыклад `user_auth_token`
- Назва функцыі: выкарыстоўвайце ніжні рэгістр camelCase (camelCase), напрыклад `userData`
- Параметры функцыі: Калі гэта функцыя зваротнага выкліку, назавіце яе ў ніжнім рэгістры, напрыклад, `onChange`
- Вызначэнні канстант на ўзроўні модуля выкарыстоўваюць стыль падкрэслівання ў верхнім рэгістры `UPPER_SNAKE_CASE`
- Не пішыце `get`, такія бессэнсоўныя прэфіксы, такія як: пішыце `cookieByHeader` замест `getCookie`
- Глабальныя канстанты/канстанты ўзроўню модуля: выкарыстоўвайце стыль падкрэслівання ў верхнім рэгістры (UPPER_SNAKE_CASE), напр. `CODE_TO_ID`, `ID_TO_LANG`

## Модульны механізм

- Імпарт: імпарт функцыянуе дакладна па патрабаванні, забараняе прамы імпарт усяго модуля (пазбягайце `import * as x` або імпартуйце вялікія аб'екты)
- Экспарт: адключыць экспарт аб'ектаў. Экспарт функцый і зменных у выглядзе адзінак, напрыклад `export const X=1, abc=()=>{};`. Паспрабуйце аб'яднаць іх з const + коска, каб аб'явіць экспартаваны кантэнт. Калі файл мае толькі адну функцыю, выкарыстоўвайце `export default`
- Калі вам не трэба выклікаць унутраную функцыю экспарту па змаўчанні, пазбягайце спачатку дэклараваць канстанты, а потым экспартаваць іх у канцы файла.
- Разрозненне шляху: пры атрыманні бягучага шляху да каталога трэба выкарыстоўваць `import.meta.dirname`
