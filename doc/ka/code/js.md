# JS კოდის სპეციფიკაცია

## კოდირების სტილი

- მარტივი, ელეგანტური და ეფექტური, ყველაზე თანამედროვე js სინტაქსის გამოყენებით
- ზედმეტად გრძელი ფუნქციების გაყოფა და ინტერფეისის დიზაინი უნდა იყოს დაბალი დაწყვილება და მაღალი შეკრულობა.
- გამოიყენეთ მასივები + მარყუჟები / თითოეულისთვის / რუქაზე მეტი და დაწერეთ ნაკლებად განმეორებადი კოდი
- არ დაწეროთ ფუნქციის კომენტარები მარტივი კოდით, რათა თავიდან აიცილოთ ზედმეტი კომენტარები. რთული ლოგიკა და სპეციალური საჭიროებები ახსნილია დახვეწილ ჩინურ ენაზე.
- განმეორებითი კოდი აბსტრაქტირებულია ფუნქციებად, რათა შემცირდეს სიჭარბე. მაგალითად, `cosnt A=new Table({ style: { border: NO_BORDER } }), B=new Table({ style: { border: NO_BORDER } })` შეიძლება დაიწეროს როგორც `const newTable=()=>new Table({ style: { border: NO_BORDER } }), A=newTable(), B=newTable()`
- ცვლადის დეკლარაცია: შეუთავსეთ რამდენიმე ზედიზედ `const` დეკლარაცია ერთ განცხადებაში. სამ სტრიქონში დაწერის ნაცვლად ჩაწერეთ `const a=1, b=2, c=3;`
- ასინქრონული დამუშავება: გამოიყენეთ `await`, აკრძალეთ `.then` ჯაჭვური ზარების გამოყენება
- არ გამოიმუშაოთ გამონაკლისების მართვის კოდი ავტომატურად, არ ჩაიწეროთ `try...catch` (`try ... catch` შენახულია ხელით, არსებული `try catch` არ უნდა წაიშალოს)
- ჯერ სუფთა ფუნქციები: ჩაწერეთ მხოლოდ სუფთა ფუნქციები, არასოდეს კლასები
- გამოიყენეთ ისრის ფუნქციები `const funcName = () => {}` და არ გამოიყენოთ `function` საკვანძო სიტყვა (გარდა გენერატორებისა); თუ შეგიძლიათ გამოიყენოთ .bind პარამეტრების დასაკავშირებლად, მოერიდეთ ფუნქციების განსაზღვრას
- კოდის ხელახალი გამოყენება: ყურადღება მიაქციეთ ხელახლა გამოყენებას, ამოიღეთ მეტი მცირე ფუნქციები და მტკიცედ მოერიდეთ მსგავსი ან კოპირებით ჩასმული კოდის სტრუქტურების დიდ რაოდენობას.
- ობიექტზე წვდომა: პრიორიტეტი მიენიჭეთ დესტრუქციული დავალების გამოყენებას საჭირო ატრიბუტების ამოსაღებად, მოერიდეთ წერტილების განმეორებით გამოყენებას შინაგანად ღრმა და ჩადგმულ ატრიბუტებზე წვდომისთვის და შერწყმა განმეორებითი არჩევითი ჯაჭვის განსჯა
- არ გამოიყენოთ ობიექტები ფუნქციის პარამეტრებისთვის, არ დაწეროთ `{a,b,c}`, ჩაწეროთ `a, b, c`; თუ ბევრი არასავალდებულო პარამეტრია, გამოიყენეთ [[კონფიგურაციის ელემენტების რაოდენობა, კონფიგურაციის ელემენტები], [კონფიგურაციის ელემენტების რაოდენობა, კონფიგურაციის ელემენტები],...] ამ პარადიგმაში, კონფიგურაციის ელემენტები განისაზღვრება რიცხვითი მუდმივებით.
- მრავალმნიშვნელოვანი დაბრუნებისთვის გამოიყენეთ მასივი `[a,b,c]` `{a,b,c}`-ის ნაცვლად. თუ 3-ზე მეტი დაბრუნების მნიშვნელობაა, გამოიყენეთ რიცხვითი მუდმივები პოზიციური მნიშვნელობის დასადგენად.
- არ არის საჭირო სტრიქონი მდგომარეობის წარმოსადგენად, გამოიყენეთ რიცხვითი მუდმივები მდგომარეობის დასადგენად
- არ არის გამოყენებული სტრიქონების შაბლონი (``), გამოიყენება სტრიქონების შეერთება (+), გარდა იმპორტისა (მოსახერხებელია Vite სტატიკური ანალიზისთვის)
- ციკლისთვის, თუ სერიული ნომერი გჭირდებათ, გამოიყენეთ `++i` `i++`-ის ნაცვლად.
- მოერიდეთ არასაჭირო`?.`-ს და აკრძალეთ თავდაცვითი პროგრამირება
- ცვლადის სახელს არ სჭირდება მრავლობითი რიცხვი და მთავრდება `_li` სიის მითითებისთვის.

## დასახელების კონვენცია

- დასახელება მინიმალიზმს მისდევს. გამოიყენეთ მოკლე, მაგრამ შინაარსიანი სახელები, მაგალითად: გამოიყენეთ `rm` `remove`, `delete`, `del`-ის ნაცვლად. თუმცა, თქვენ ასევე უნდა მოერიდოთ უკიდურესობებს, მაგალითად: არ გამოიყენოთ ერთი ასო `m` `map`-ის ნაცვლად.
- შეეცადეთ გამოიყენოთ მხოლოდ ზმნები ფუნქციების დასახელებაში. თუ თქვენ შეგიძლიათ გამოიყენოთ ერთი სიტყვა მის გამოსახატავად, ნუ გამოიყენებთ ორ სიტყვას. არსებითი სახელები აისახება ფაილის სახელებში. თუ თქვენ გჭირდებათ ზმნის დამატება ფაილის სახელზე, გთხოვთ, ჯერ დააყენოთ არსებითი სახელი და ბოლოს ზმნა. მაგალითად: `profileSet.js` ნაცვლად `setProfile.js`-ისა
- ცვლადის სახელი: გამოიყენეთ ხაზგასმული სტილი (snake_case), მაგალითად `user_auth_token`; თუ ცვლადი არის ფუნქცია, გამოიყენეთ პატარა რეგისტრირებული აქლემის დასახელების სტილი
- ფუნქციის სახელი: გამოიყენეთ მცირე რეგისტრი camelCase (camelCase), მაგალითად `userData`
- ფუნქციის პარამეტრები: თუ ეს არის გამოძახების ფუნქცია, დაასახელეთ იგი მცირე ასოებით, როგორიცაა `onChange`
- მოდულის დონის მუდმივი განსაზღვრებები იყენებს ხაზგასმის სტილს დიდი ასოებით `UPPER_SNAKE_CASE`
- არ დაწეროთ `get` ისეთი უაზრო პრეფიქსები, როგორიცაა: `getCookie`-ის ნაცვლად ჩაწერეთ `cookieByHeader`
- გლობალური/მოდულის დონის მუდმივები: გამოიყენეთ დიდი რეგისტრის სტილი (UPPER_SNAKE_CASE), მაგ. `CODE_TO_ID`, `ID_TO_LANG`

## მოდულური მექანიზმი

- იმპორტი: იმპორტი ფუნქციონირებს ზუსტად მოთხოვნისამებრ, აკრძალეთ მთელი მოდულის პირდაპირი იმპორტი (აარიდეთ `import * as x` ან დიდი ობიექტების იმპორტს)
- ექსპორტი: გამორთეთ ობიექტების ექსპორტი. ფუნქციების და ცვლადების ექსპორტი ერთეულებად, როგორიცაა `export const X=1, abc=()=>{};`. შეეცადეთ დააკავშიროთ ისინი const + მძიმით, რათა გამოაცხადოთ ექსპორტირებული შინაარსი. თუ ფაილს აქვს მხოლოდ ერთი ფუნქცია, გამოიყენეთ `export default`
- თუ არ გჭირდებათ ექსპორტის ნაგულისხმევი ფუნქციის შიგნიდან გამოძახება, თავიდან აიცილეთ მუდმივების დეკლარირება და შემდეგ მათი ექსპორტი ფაილის ბოლოს.
- ბილიკის გარჩევადობა: მიმდინარე დირექტორიაში ბილიკის მიღებისას უნდა იქნას გამოყენებული `import.meta.dirname`
