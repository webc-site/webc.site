# WebC.site

## นี่อะไรน่ะ?

### นี่คือไลบรารีองค์ประกอบเว็บดั้งเดิม

พัฒนาขึ้นโดยใช้ [ส่วนประกอบของเว็บ](https://developer.mozilla.org/docs/Web/API/Web_components) เป็นรันไทม์อิสระและเหมาะสำหรับทุกเฟรมเวิร์ก

ปรับให้เหมาะสมที่สุดสำหรับขนาด (หนึ่งในสิบของขนาดของไลบรารียอดนิยมที่คล้ายคลึงกัน)

รองรับ 75 ภาษา คุณสามารถนำเข้า js และ css เพื่อใช้งานออนไลน์ได้โดยตรง หรือเพิ่มส่วนประกอบให้กับโปรเจ็กต์ของคุณตามที่จำเป็นสำหรับการพัฒนารอง

### นี่คือกรอบการพัฒนาไลบรารีส่วนประกอบ

นี่เป็นเวอร์ชันบล็อกคงที่ของ [หนังสือนิทาน](https://storybook.js.org)

การสร้างไลบรารีส่วนประกอบสามารถปรับปรุงประสิทธิภาพการพัฒนาภายในของบริษัท และสร้างภาพลักษณ์ของแบรนด์ในแวดวงเทคนิคได้

ตาม [webc.site](https://github.com/webc-site/webc.site) ทีมสามารถสร้างไลบรารีส่วนประกอบของตนเองได้อย่างง่ายดายและเผยแพร่ไปยัง [หน้า github](https://pages.github.com), [หน้า cloudflare](https://pages.cloudflare.com) ฯลฯ ส่วนประกอบที่เผยแพร่สามารถแก้ไขแบบออนไลน์ได้ที่ [codepen.io](https://codepen.io) และ [jsfiddle](https://jsfiddle.net)

ในเวลาเดียวกัน กรอบงานการแปลในตัวสามารถเข้าใจถึงความเป็นสากลของส่วนประกอบและเอกสารได้อย่างรวดเร็ว เพื่อให้ผู้ชมผลิตภัณฑ์ของคุณไม่ถูกผูกมัดด้วยภาษาอีกต่อไป

### นี่คือกระบวนทัศน์ใหม่ของการพัฒนาส่วนหน้าสำหรับยุคปัญญาประดิษฐ์

เฟรมเวิร์กมีอยู่ภายใน `.agents/skills` ช่วยให้ปัญญาประดิษฐ์ช่วยคุณพัฒนาส่วนประกอบใหม่ได้ด้วยคลิกเดียวตามแนวทางปฏิบัติที่ดีที่สุด

<img src="https://i-01.eu.org/20260604121408079.avif" width="400">

ปล่อยให้โค้ดส่วนหน้าเปลี่ยนจากการเขียนโปรแกรมบรรยากาศภูเขาขยะที่ไม่ยั่งยืนไปเป็นส่วนมาตรฐานที่สามารถรักษาและพัฒนาซ้ำๆ ได้ในระยะยาว

เมื่อพิจารณาถึงลักษณะของการพัฒนาที่ได้รับความช่วยเหลือจากปัญญาประดิษฐ์ เราได้เสนอและออกแบบกระบวนทัศน์การพัฒนาส่วนหน้าใหม่:

1. **ทุกอย่างเป็นส่วนประกอบของเว็บ**
2. **ส่วนประกอบต้องเป็นอิสระจากลิงก์ข้อมูลส่วนหลัง และอินเทอร์เฟซข้อมูลถูกเปิดเผยเป็นฟังก์ชันเรียกกลับ**

ด้วยวิธีนี้ ปัญญาประดิษฐ์สามารถสร้างข้อมูลปลอมและแสดงสถานะต่างๆ ของส่วนประกอบบนหน้าสาธิตได้

การแยกการพัฒนาส่วนประกอบและอินเทอร์เฟซข้อมูล รวมกับ [กรอบงานส่วนประกอบ webc.site](https://github.com/webc-site/webc.site) คุณสมบัติที่สำคัญอีกประการหนึ่ง:

**แต่ละองค์ประกอบสามารถเริ่มต้นเซิร์ฟเวอร์การพัฒนาได้อย่างอิสระ และพัฒนาและแก้ไขจุดบกพร่องในหน้าสาธิต**

<img src="https://i-01.eu.org/webcSiteDemo.svg" width="400">
<img src="https://i-01.eu.org/20260604112614499.avif" width="400">

ซึ่งจะช่วยลดลิงก์ส่วนหน้าสำหรับการพัฒนาและการดีบักของแต่ละองค์ประกอบเป็น `0`

ด้วยวิธีนี้ เครื่องมือพัฒนาที่ได้รับความช่วยเหลือจากปัญญาประดิษฐ์ ([Claude Code](https://code.claude.com), [OpenAI Codex](https://openai.com), [Google Antigravity](https://antigravity.google), [Cursor](https://cursor.com), [Windsurf](https://windsurf.com), [Devin](https://devin.ai), [OpenHands](https://github.com/All-Hands-AI/OpenHands) ฯลฯ) คุณสามารถเพลิดเพลินกับการพัฒนาอัตโนมัติเต็มรูปแบบ การดีบักเบราว์เซอร์แบบเปิด และไม่ถูกบล็อกโดยปัจจัยที่ยุ่งวุ่นวายต่างๆ อีกต่อไป (ข้อมูลแบ็กเอนด์ สถานะการเข้าสู่ระบบ เส้นทาง การกำหนดเส้นทาง ฯลฯ)

หากคุณต้องการควบคุมปัญญาประดิษฐ์ ลดต้นทุน และเพิ่มประสิทธิภาพ คุณต้องมีกระบวนทัศน์ใหม่ที่ลดสภาพแวดล้อมการพัฒนาและการแก้ไขจุดบกพร่องของแต่ละส่วนประกอบให้เหลือน้อยที่สุด

### ไลบรารีส่วนประกอบของเว็บ

#### การเพิ่มประสิทธิภาพขนาดขั้นสูงสุด

ตัวอย่างเช่น สำหรับส่วนประกอบแถบเลื่อนเสมือน ให้เปรียบเทียบขนาดของสคริปต์และสไตล์ [OverlayScrollbars](https://github.com/KingSora/OverlayScrollbars) ยอดนิยมที่อยู่หลัง `gz` ดังที่แสดงในตารางต่อไปนี้:

|                   |   js    |  css   |     total      |
| :---------------- | :-----: | :----: | :------------: |
| overlayscrollbars | 14520 B | 2584 B | 17104 B (9.1X) |
| Scroll            | 1159 B  | 721 B  |   ⭐ 1880 บ    |

ส่วนประกอบแถบเลื่อนเสมือนของ `webc.site` เป็นเพียง `OverlayScrollbars` `10%` เท่านั้น

#### รันไทม์เป็นอิสระ

ไลบรารีคอมโพเนนต์ webc.site สร้างขึ้นตามมาตรฐาน [Web Components](https://developer.mozilla.org/docs/Web/API/Web_components) โดยธรรมชาติแล้วจะไม่เชื่อเรื่องรันไทม์ ไม่ว่าคุณจะใช้ React, Vue, Angular, Svelte หรือโปรเจ็กต์ HTML/JS แบบดั้งเดิม คอมโพเนนต์ทั้งหมดสามารถทำงานได้โดยตรงในเบราว์เซอร์ เช่น แท็ก HTML ดั้งเดิม โดยไม่จำเป็นต้องใช้แพ็คเกจรองสำหรับเฟรมเวิร์กเฉพาะ

##### เปรียบเทียบกับ shacn

[shadcn/ui](https://ui.shadcn.com) ใช้โหมดการกระจายโค้ด ซึ่งกำหนดให้คุณต้องคัดลอกโค้ดส่วนประกอบลงในโปรเจ็กต์ของคุณเองโดยตรง มีความผูกพันอย่างแน่นหนากับระบบนิเวศ React และกลุ่มเครื่องมือก่อสร้างเฉพาะ (เช่น Tailwind CSS) และไม่สามารถใช้ข้ามเฟรมเวิร์กได้ นอกจากนี้ยังมีข้อกำหนดบางประการสำหรับการกำหนดค่าทางวิศวกรรมของโครงการอีกด้วย

และองค์ประกอบของ `webc.site`:
- **ข้ามเฟรมเวิร์กสากล**: เขียนเพียงครั้งเดียวและสามารถนำไปใช้และเรียกใช้โดยตรงในเฟรมเวิร์กฟรอนต์เอนด์ใดก็ได้ (แม้แต่สภาพแวดล้อมที่ไม่มีเฟรมเวิร์ก)
- **การพึ่งพาสภาพแวดล้อมเป็นศูนย์**: ไม่ต้องใช้เครื่องมือบรรจุภัณฑ์เฉพาะหรือตัวประมวลผลล่วงหน้า CSS เพียงนำเข้าไฟล์คงที่ที่คอมไพล์แล้วก็สามารถใช้งานได้ทันที

## การพัฒนาและการแก้ไขอัตโนมัติ

### การพัฒนาและการดีบักในท้องถิ่น

แต่ละองค์ประกอบของหน้าเว็บรองรับการเริ่มต้นเซิร์ฟเวอร์การพัฒนาแยกจากกัน โดยให้สภาพแวดล้อมการพัฒนาและการดีบักน้อยที่สุด:

- **สร้างส่วนประกอบใหม่**: เรียกใช้ `./sh/new.js [css|js] [ชื่อส่วนประกอบ]` เพื่อสร้างโครงกระดูกการพัฒนาส่วนประกอบอย่างรวดเร็ว (รวมถึงหน้าสาธิตและการกำหนดค่าสากล)
- **การดีบักแบบอิสระ**: เรียกใช้ `./dev.js [ชื่อส่วนประกอบ]` (หรือเรียกใช้ `./dev.js` โดยตรง) เพื่อเริ่มเซิร์ฟเวอร์การพัฒนา Vite และเปิดหน้าสาธิตของส่วนประกอบในเบราว์เซอร์โดยอัตโนมัติ รองรับการอัพเดตคอมโพเนนต์ร้อน (HMR) เนื่องจากลิงก์การพัฒนาและลิงก์ข้อมูลแยกจากกัน จึงไม่มีลิงก์ส่วนหน้าสำหรับการพัฒนาซึ่งเหมาะมากสำหรับ AI (เช่น Claude Code, Google Antigravity ฯลฯ) เพื่อช่วยในการพัฒนาอัตโนมัติเต็มรูปแบบและการดีบักเบราว์เซอร์

### เผยแพร่ไลบรารีส่วนประกอบ

#### webc dist

กระบวนการบรรจุและเผยแพร่เป็นไปโดยอัตโนมัติอย่างมาก:
- **การบรรจุทรัพยากรแบบคงที่**: การรัน `./sh/dist.js` (นั่นคือ การดำเนินการงาน `webc dist`) สามารถสแกนส่วนประกอบของหน้าเว็บทั้งหมดภายใต้ `src/` ได้โดยอัตโนมัติ คอมไพล์ บีบอัด และส่งออกไปยังไดเร็กทอรี `dist/` ไดเร็กทอรีนี้มีไฟล์ JS/CSS สำหรับการเผยแพร่ที่ใช้งานจริง ไซต์สาธิตที่พร้อมสำหรับการปรับใช้กับ GitHub Pages หรือ Cloudflare Pages และสร้างไฟล์การกำหนดเส้นทาง `_redirects` และ `404.html` โดยอัตโนมัติ
- **การเผยแพร่ NPM**: เรียกใช้ `./sh/npmDist.js` สคริปต์จะจัดการการขึ้นต่อกันของส่วนประกอบโดยอัตโนมัติ เพิ่มหมายเลขเวอร์ชันของแพ็คเกจ และดำเนินการ `npm publish` ได้ในคลิกเดียวหลังจากแพ็กเกจเสร็จสมบูรณ์เพื่อเผยแพร่ไลบรารีส่วนประกอบไปยัง NPM

### แพลตฟอร์มมีหน้าที่รับผิดชอบในการแปล

กรอบงานมีกระบวนการแปลอัตโนมัติในตัว:
1. กำหนดค่ากฎการแปล (เช่น ภาษาต้นฉบับ ภาษาเป้าหมาย และไดเร็กทอรีที่จะแปล) ใน `tran.yml` ในไดเร็กทอรีรากของโปรเจ็กต์
2. ใช้ตัวยึดตำแหน่งในส่วนประกอบหรือเอกสารโดยไม่ต้องฮาร์ดโค้ดหลายภาษาในโค้ดของคุณ
3. เรียกใช้ `./sh/tran.js` และเครื่องมือการแปลจะแยกข้อความโดยอัตโนมัติและเรียกใช้บริการแปล WebC เพื่อประมวลผล สร้างรายการแปล YAML และเอกสาร README ในภาษาต่างๆ โดยอัตโนมัติ

### ตัวแปรสภาพแวดล้อม

เมื่อดำเนินการแปลอัตโนมัติ คุณจะต้องกำหนดค่าตัวแปรสภาพแวดล้อมต่อไปนี้เพื่อให้สิทธิ์การเข้าถึง API การแปล:

- **`WEBC_SITE_TOKEN`** (หรือ **`WEBC_TOKEN`**): คีย์การตรวจสอบสิทธิ์ที่ใช้ในการเข้าถึง `webc.site` API การแปลอัตโนมัติ สามารถฉีดเป็นตัวแปรสภาพแวดล้อมหรือเขียน `token` ในไฟล์การกำหนดค่า `~/.config/webc.site.yml` ในโฮมไดเร็กทอรีของผู้ใช้ภายในเครื่อง
